`mgr-storage-server` 和 `mgr-storage-proxy` 是 {productname} 5.0 及更高版本附带的辅助脚本。
endif::[]

ifeval::[{uyuni-content} == true]

`mgr-storage-server` 和 `mgr-storage-proxy` 是 {productname} 附带的辅助脚本。
endif::[]

它们旨在为 {productname} 服务器和代理配置存储。

这些脚本将磁盘设备作为参数。`mgr-storage-proxy` 需要一个存储磁盘设备的单个参数。`mgr-storage-server` 需要一个存储磁盘设备，并可以选择性地接受第二个参数，作为专用数据库磁盘设备。
虽然普通存储和数据库存储可以位于同一磁盘上，但建议将数据库放在专用的高性能磁盘上，以确保更好的性能和更容易的管理。


== 这些工具的功能


`mgr-storage-server` 和 `mgr-storage-proxy` 均执行标准存储设置操作：

* 验证提供的存储设备。
* 确保设备为空且适合使用。
* 在指定设备上创建 XFS 文件系统。
* 临时挂载设备以进行数据迁移。
* 将相关存储目录迁移到新设备。
* 在 `/etc/fstab` 中创建条目，以便存储在启动时自动挂载。
* 将设备重新挂载到最终位置。


.工具的专属特性
[cols="1,3a"]
|===
| `mgr-storage-server`
|
选择性支持数据库存储专用设备。

在迁移期间停止 {productname} 服务，迁移完成后重启服务。

将 Podman 卷目录 `/var/lib/containers/storage/volumes` 移动到准备好的存储中，并可选择将 `/var/lib/containers/storage/volumes/var-pgsql` 移动到准备好的数据库存储中。

| `mgr-storage-proxy`
|
仅专注于设置代理存储设备（不支持设置数据库存储设备）。

迁移期间停止并重启代理服务。

将 podman 卷目录 `/var/lib/containers/storage/volumes` 移动到准备好的存储中。

|===


[NOTE]
====
这两个工具自动化标准 Linux 存储操作。
除了 Linux 管理员手动执行的操作外，不存在任何隐藏或自定义逻辑。
====


== 这些工具*不*执行的功能

* 它们\\**不会**创建或管理 LVM 卷。
* 它们*不会*配置独立磁盘冗余阵列或复杂存储拓扑。
* 它们\\**不会**阻止您在完成设置后使用常规 Linux 工具管理存储。
* 它们*不会*提供动态调整大小或展开功能 — 这些操作必须通过标准 Linux 存储工具完成。


== 容器管理

完成存储配置后，您可通过标准 Linux 命令安全地管理存储设备。


=== 示例

.示例 1：扩展存储（如果使用 LVM）

----
lvextend -L +10G /dev/your_vg/your_lv
xfs_growfs /var/lib/containers/storage/volumes
----


.示例 2：迁移到更大的磁盘 

. 添加并格式化新磁盘。
. 临时挂载新磁盘。
. 使用 `rsync` 复制数据。
. 更新 `/etc/fstab`。
. 将新磁盘重新挂载到适当位置。


== 适用与不适用场景

[IMPORTANT]
====
更改存储设置前，务必创建备份。
====

* 在初始存储设置期间或当迁移到需要工具处理数据迁移和更新 `/etc/fstab` 的新存储设备时，仅使用这些工具 *only*。
* *不*要重新运行这些脚本以调整或展开存储。使用标准 Linux 工具（例如 `lvextend`、`xfs_growfs`）进行此类操作。


== 摘要

`mgr-storage-server` 和 `mgr-storage-proxy` 有助于使用标准的 Linux 存储实践为 {productname} 组件自动执行初始持久存储设置。
它们不会限制或干扰后续的标准存储管理。

完成设置后，您可继续使用熟悉的 Linux 工具管理存储设备。

[IMPORTANT]
====
完整的数据库卷可能会对系统操作造成重大问题。
由于磁盘使用通知尚未适应容器化环境，建议用户通过 Grafana、Prometheus 或任何其他首选方法监控 Podman 卷本身使用的磁盘空间。
请特别注意位于 `/var/lib/containers/storage/volumes/` 下的 var-pgsql 卷。
====
