本节包含有关注册运行 {sll} 操作系统的客户端的信息。{sll} 客户端基于 {rhel} 或 {centos}。


[NOTE]
====
{sll} 客户端有时也称为 {sleses} (SLESES)、Liberty、RES 或 {redhat} {es}。
====

{sll} 提供的软件通道{suse}仅提供软件包的更新，不提供软件包本身。
要注册 {sll} 客户端，您需要注册 {sll} 产品（如下所述）以创建必要的基础通道，然后将任何所需的 {redhat} 或 {centos} 软件包作为自定义子通道导入。
在您可以应用 {sll} 软件通道提供的更新之前，您必须直接从 {redhat} 或 {centos} 获取初始软件包。

== 添加软件通道


对于 {sll} 客户端，某些所需的软件包包含在 {rhel} 或 {centos} 安装媒体中。
在您可以注册 {sll} 客户端之前，必须安装这些软件包。

{sll} 产品由 {scclongform} 提供。
这也包括客户端工具软件包。

在将 {sll} 客户端注册到您的 {productname} 服务器之前，您需要添加所需的软件通道，并同步这些通道。

您需要选择两组不同的通道，一组用于 {sll}，另一组用于客户端工具。

您需要一个与正确的 {sll} 通道关联的激活密钥。
有关激活密钥的更多信息，请参见 xref:client-configuration:activation-keys.adoc[]。

[[add.base.media]]
=== 添加基础媒体

{sll} 软件通道仅提供更新，而不提供软件包本身。
要注册 {sll} 客户端，首先注册 {sll} 产品（如下所述）以创建基础通道，然后将必要的 Red Hat 或 CentOS 软件包导入为自定义子通道。
初始软件包必须直接从 {redhat} 或 {centos} 获取，然后才能应用来自 {sll} 的更新。
重要的是，您不需要维护您的 Red Hat 订阅，但请咨询您的法律部门以检查在迁移期间是否有任何对 Red Hat 的持续付款义务。
为了确保您拥有所需的所有软件包，请使用完整的 DVD 映像，而不是最小或 {jeos} 映像。

您可以使用 {productname} 自定义通道来设置 {rhel} 或 {centos} 媒体。
基础媒体上的所有软件包必须镜像到子通道中。

您可以自由选择通道名称。

[IMPORTANT]
====
对于 {redhat} 10、9 和 8 客户端，请添加 Base 和 AppStream 通道。
您需要从这两个通道获取软件包。
如果您不添加这两个通道，由于缺少软件包，您将无法创建启动储存库。
====

如果您使用的是模块化通道，您必须在客户端上启用 Python 3.6 模块流。
如果您不提供 Python 3.6，则 `spacecmd` 软件包的安装将失败。


.过程：升级将基础媒体添加到自定义通道（RHEL 7.0 示例）
[role=procedure]
____
[WARNING]
====
您必须使用来自 {redhat} 的原始安装媒体。
您必须使用主要版本发布（例如 `7`），而不是次要版本发布（例如 `7.9`）。
====
. 将基础媒体映像复制到 {productname} 服务器容器主机上的 `/tmp/` 目录中。

. 要将基础媒体映像添加到 {productname} 服务器容器，请使用：

+
----
mgradm distribution copy /tmp/rhel-server-7.0-x86_64-dvd.iso rhel-server-7.0
----

+
容器中的目标将是 `/srv/www/distributions/rhel-server-7.0`。

. 将软件包导入您之前创建的子通道：

+

[source,shell]
----
mgrctl exec -- spacewalk-repo-sync -c <channel-label> -u file:///srv/www/distributions/rhel-server-7.0/
----

____

对于版本 9 和 8 客户端，请相应调整标签和目录，此外还需同步基础通道和 AppStream 通道：

[source,shell]
----
mgrctl exec -- spacewalk-repo-sync -c <channel-label> -u file:///srv/www/distributions/rhel-server-9.0/BaseOS
mgrctl exec -- spacewalk-repo-sync -c <channel-label> -u file:///srv/www/distributions/rhel-server-9.0/AppStream
----

或：

[source,shell]
----
mgrctl exec -- spacewalk-repo-sync -c <channel-label> -u file:///srv/www/distributions/rhel-server-8.0/BaseOS
mgrctl exec -- spacewalk-repo-sync -c <channel-label> -u file:///srv/www/distributions/rhel-server-8.0/AppStream
----

[[clients-sleses-el-migration]]
== 将 Enterprise Linux (EL) 客户端迁移到 {sll}

如果某个 Enterprise Linux (EL) 客户端（例如 {rhel} 和 {centos}）已在 {productname} 上注册为受控端，且用户想要将其迁移到 {sll}，则可以使用重新激活密钥来应用用于驱动迁移的激活密钥。

有关重新激活密钥的更多信息，请参见 xref:client-configuration:activation-keys.adoc#activation-keys-reactivation[]。

重新激活密钥是针对每个受控端的，可以通过 {webui} 生成或使用 API。
有关更多信息，请参见 https://documentation.suse.com/multi-linux-manager/5.1/api/suse-manager/api/system.html#apidoc-system-obtainReactivationKey-loggedInUser-sid.

要重新激活客户端，用户可以在客户端上运行启动脚本，并以环境变量的形式传递重新激活密钥。
示例：

----
REACTIVATION_KEY=<KEY> ./bootstrap_liberate9.sh
----

另一种方法是在位于 {salt}（或 `/etc/venv-salt-minon/minion.d/susemanager.conf`）的 `/etc/salt-minon/minion.d/susemanager.conf` 客户端配置文件中添加一些特殊标志，内容如下（将此内容与现有内容合并）：

----
grains:
    susemanager:
        activation_key: "<KEY_ID>"
        management_key: "MINION_REACTIVATION_KEY"
----

更改 `susemanager.conf` 文件后，需要在 `salt-minion` 服务器上重启 {salt} 服务。 默认情况下，

----
systemctl restart venv-salt-minon
----

或对于遗留的 {salt}，使用：

----
systemctl restart salt-minon
----
