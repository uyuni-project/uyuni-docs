== 이미지 빌드 개요


{productname}을(를) 사용하면 시스템 관리자가 컨테이너와 OS 이미지를 빌드하고 결과를 이미지 저장소에 푸시할 수 있습니다.

.절차 이미지 빌드 및 푸시
[role=procedure]
_____

. 이미지 저장소를 정의합니다.

. 이미지 프로파일을 정의하고 이를 소스(git 리포지토리 또는 디렉토리)와 연결합니다.

. 이미지를 빌드합니다.

. 이미지를 이미지 저장소로 푸시합니다.
_____

{productname}은(는) 다음 두 가지 빌드 유형을 지원합니다: Dockerfile 및 Kiwi 빌드 유형입니다.
Kiwi 빌드 유형은 시스템, 가상 및 기타 이미지를 빌드하는 데 사용됩니다.

ifeval::[{mlm-content} == true]

[[at.images.docker.requirements]]
=== 요구사항

컨테이너 기능은 {salt} 클라이언트에서 {sles} 12 이상을 실행할 때 사용할 수 있습니다.
시작하기 전에 환경이 다음 요구 사항을 충족하는지 확인하십시오:

* Dockerfile 및 구성 스크립트를 포함하는 게시된 git 리포지토리입니다.
  리포지토리는 공개 또는 비공개일 수 있으며, GitHub, GitLab 또는 BitBucket에 호스팅되어야 합니다.
* Docker 레지스트리와 같이 올바르게 구성된 이미지 저장소입니다.

컨테이너에 대한 자세한 내용은 https://documentation.suse.com/container/all/html/Container-guide/.을(를) 참조하십시오.


[[at.images.docker.buildhost]]
=== 빌드 호스트 생성

{productname}으로 이미지를 빌드하려면 빌드 호스트를 생성하고 구성해야 합니다.
컨테이너 빌드 호스트는 {salt} 클라이언트에서 {sle} 12 이상을 실행할 때 사용됩니다.
이 섹션은 빌드 호스트의 초기 구성을 안내합니다.

[IMPORTANT]
====
빌드 호스트의 운영 체제는 대상 이미지의 운영 체제와 일치해야 합니다.

예를 들어, {sles}{nbsp}15 (SP2 이상) OS 버전을 실행하는 빌드 호스트에서 {sles}{nbsp}15 기반 이미지를 빌드합니다.
{sles}{nbsp}12 SP5 또는 {sles}{nbsp}12 SP4 OS 버전을 실행하는 빌드 호스트에서 {sles}{nbsp}12 기반 이미지를 빌드합니다.

아키텍처 간 빌드는 지원되지 않습니다.
////
2022-05-12, ke: at least for now, according to feedback, this is unsupported:
For example, you must build Raspberry Pi {sles}{nbsp}15 SP3 images on a Raspberry Pi (aarch64 architecture) build host running {sles}{nbsp}15 SP3.
////
====

{productname} {webui}에서 다음 단계를 수행하여 빌드 호스트를 구성합니다:

.절차 호스트 빌드
[role=procedure]
_____

. {salt} 클라이언트를 menu:시스템[시스템] 개요 페이지에서 빌드 호스트로 지정합니다.

. 선택한 클라이언트의 `System Details` 페이지에서 컨테이너 모듈을 할당합니다.
  menu:소프트웨어[소프트웨어 채널](으)로 이동하여 컨테이너 모듈을 활성화합니다(예: `SLE-Module-Containers15-Pool` 및 `SLE-Module-Containers15-Updates`).
  계속 진행하려면 btn:[다음]을 클릭합니다.

. `소프트웨어 채널 변경`을 예약하고 btn:[확인]을 클릭합니다.

. `System Details` 탭에서 menu:속성[] 페이지를 선택하고 `Add-on System Types` 목록에서 `Container Build Host`을(를) 활성화합니다.
  btn:[속성 업데이트]를 클릭하여 확인합니다.

. `Highstate`을(를) 적용하여 필요한 모든 패키지를 설치합니다.
  시스템 세부정보 탭에서 menu:상태[하이 스테이트]를 선택하고 btn:[하이 스테이트 적용]을 클릭합니다.
  또는 {productname} 서버 명령줄에서 `Highstate`을(를) 적용합니다:

+

[source,shell]
----
salt '$your_client' state.highstate
----

[[at.images.docker.buildchannels]]
=== 컨테이너에 대한 활성화 키 생성

{productname}으로 빌드된 컨테이너는 이미지를 빌드할 때 활성화 키와 연결된 채널을 리포지토리로 사용합니다.
이 섹션에서는 이 목적을 위한 임시 활성화 키를 만드는 방법을 안내합니다.

[NOTE]
====
컨테이너를 빌드하려면 `SUSE Manager Default` 이외의 채널과 연결된 활성화 키가 필요합니다.
====

.절차 활성화 키 생성
[role=procedure]
_____

. menu:시스템[활성화 키]를 선택합니다.

. btn:[키 생성]를 클릭합니다.

. `Description` 및 `Key` 이름을 입력합니다.
  드롭다운 메뉴를 사용하여 이 키와 연결할 `Base Channel`을 선택합니다.

. btn:[활성화 키 생성]으로 확인합니다.

_____

자세한 내용은 xref:client-configuration:activation-keys.adoc[]을 참조하십시오.


[[at.images.docker.imagestore]]
=== 이미지 저장소 생성

모든 빌드된 이미지는 이미지 저장소에 푸시됩니다.
이 섹션에는 이미지 저장소 생성에 대한 정보가 포함되어 있습니다.
이미지 저장소는 일반적으로 레지스트리로 참조됩니다.


.절차 이미지 저장소 생성
[role=procedure]
_____

. menu:이미지[저장소]를 선택합니다.

. `Create`을 클릭하여 새 저장소를 생성합니다.

. `Store Type`에서 올바른 유형을 선택합니다.

. `Label` 필드에 이미지 저장소의 이름을 입력합니다.

. 컨테이너 레지스트리 호스트(내부 또는 외부)의 FQDN(정규화된 도메인 이름)으로 `URI` 필드를 입력하여 이미지 레지스트리 경로를 제공합니다.

+

----
registry.example.com
----

+

레지스트리 URI를 사용하여 이미 사용 중인 레지스트리의 이미지 저장소를 지정할 수도 있습니다.

+

----
registry.example.com:5000/myregistry/myproject
----

. btn:[생성]을 클릭하여 새 이미지 저장소를 추가합니다.

_____



[[at.images.docker.profile]]
=== 이미지 프로파일 생성

모든 컨테이너 이미지는 빌딩 지침을 포함하는 이미지 프로파일을 사용하여 빌드됩니다.
이 섹션에는 {productname} {webui}를 사용하여 이미지 프로파일을 생성하는 방법에 대한 정보가 포함되어 있습니다.

.절차 이미지 프로파일 생성
[role=procedure]
_____

. 이미지 프로파일을 생성하려면 menu:이미지[프로파일]을 선택하고 btn:[생성]을 클릭합니다.

. `Label` 필드를 작성하여 이미지 프로파일의 이름을 입력합니다.

+

[NOTE]
====
컨테이너 이미지 태그의 형식이 `myproject/myimage`과 같은 경우, 이미지 저장소 레지스트리 URI에 `/myproject` 접미사가 포함되어 있는지 확인합니다.
====

. `Dockerfile`을 `Image Type`(으)로 사용합니다.

. 드롭다운 메뉴를 사용하여 `Target Image Store` 필드에서 레지스트리를 선택합니다.

. `Path` 필드에 GitHub, GitLab 또는 BitBucket 리포지토리 URL을 입력합니다.
  경로는 빌드 호스트의 로컬 디렉토리일 수도 있습니다.
  URL은 `http`, `https` 또는 토큰 인증 URL이어야 합니다.
  GitHub 또는 GitLab의 경우 다음 형식 중 하나를 사용하십시오:

+

.GitHub 경로 옵션

* GitHub 단일 사용자 프로젝트 리포지토리

+
----
https://github.com/USER/project.git#branchname:folder
----

+

* GitHub 조직 프로젝트 리포지토리

+
----
https://github.com/ORG/project.git#branchname:folder
----

+

* GitHub 토큰 인증

+

git 리포지토리가 비공개인 경우 프로파일의 URL을 수정하여 인증을 포함해야 합니다.
GitHub 토큰으로 인증하기 위해 이 URL 형식을 사용하십시오:

+

----
https://USER:<AUTHENTICATION_TOKEN>@github.com/USER/project.git#master:/container/
----

+

.GitLab 경로 옵션

* GitLab 단일 사용자 프로젝트 리포지토리

+

----
https://gitlab.example.com/USER/project.git#master:/container/
----

* GitLab 그룹 프로젝트 리포지토리

+

----
https://gitlab.example.com/GROUP/project.git#master:/container/
----

* GitLab 토큰 인증

+

git 리포지토리가 비공개이고 공개적으로 접근할 수 없는 경우, 프로파일의 git URL을 수정하여 인증을 포함해야 합니다.
GitLab 토큰으로 인증하기 위해 이 URL 형식을 사용하십시오:

+

----
https://gitlab-ci-token:<AUTHENTICATION_TOKEN>@gitlab.example.com/USER/project.git#master:/container/
----

+

[IMPORTANT]
====
git 브랜치를 지정하지 않으면 기본적으로 `master` 브랜치가 사용됩니다.
`folder`가 지정되지 않으면 이미지 소스(Dockerfile 소스)는 GitHub 또는 GitLab 체크아웃의 루트 디렉토리에 있어야 합니다.
====

. `Activation Key`을 선택하십시오.
  활성화 키는 프로파일을 사용하는 이미지가 올바른 채널 및 패키지에 할당되도록 보장합니다.

+

[NOTE]
====
활성화 키를 이미지 프로파일과 연결하면 프로파일을 사용하는 모든 이미지가 올바른 소프트웨어 채널과 채널의 모든 패키지를 사용하도록 할 수 있습니다.
====

. btn:[생성] 버튼을 클릭하십시오.

_____


[[at.images.docker.sourceexample]]
==== 예시 Dockerfile 소스

재사용할 수 있는 이미지 프로파일이 https://github.com/SUSE/manager-build-profiles.에 게시됩니다.

[NOTE]
====
`ARG` 매개변수는 빌드된 이미지가 {productname}에서 제공하는 원하는 리포지토리와 연결되도록 보장합니다.
`ARG` 매개변수는 또한 빌드 호스트 자체에서 사용되는 {sles} 버전과 다를 수 있는 {sles}의 이미지 버전을 빌드할 수 있게 해줍니다.

예: `ARG repo` 매개변수와 리포지토리 파일을 가리키는 `echo` 명령은 원하는 채널 버전에 대한 리포지토리 파일에 올바른 경로를 생성한 후 삽입합니다.

리포지토리는 이미지 프로파일에 할당한 활성화 키로 결정됩니다.
====

----
FROM registry.example.com/sles12sp2
MAINTAINER Tux Administrator "tux@example.com"

### Begin: These lines are required for use with {productname}

ARG repo
ARG cert

# Add the correct certificate
RUN echo "$cert" > /etc/pki/trust/anchors/RHN-ORG-TRUSTED-SSL-CERT.pem

# Update certificate trust store
RUN update-ca-certificates

# Add the repository path to the image
RUN echo "$repo" > /etc/zypp/repos.d/susemanager:dockerbuild.repo

### End: These lines are required for use with {productname}

# Add the package script
ADD add_packages.sh /root/add_packages.sh

# Run the package script
RUN /root/add_packages.sh

# After building remove the repository path from image
RUN rm -f /etc/zypp/repos.d/susemanager:dockerbuild.repo
----

==== 사용자 정의 정보 키-값 쌍을 Docker `buildargs`로 사용

사용자 정의 정보 키-값 쌍을 할당하여 이미지 프로파일에 정보를 첨부할 수 있습니다.
또한, 이러한 키-값 쌍은 `buildargs`으로 Docker 빌드 명령에 전달됩니다.

사용 가능한 사용자 정의 정보 키 및 추가 키 생성에 대한 자세한 정보는 xref:reference:systems/custom-system-info.adoc[]을 참조하십시오.

=== 이미지 빌드


이미지를 빌드하는 방법은 두 가지가 있습니다.
첫 번째 방법은 처음부터 만드는 것입니다. 
이를 위해 왼쪽 탐색 바에서 menu:이미지[빌드]를 선택하거나 menu:이미지[프로파일] 목록에서 빌드 아이콘을 클릭하고 절차를 따르십시오.

.절차 이미지를 빌드하는 중입니다.
[role=procedure]
_____

. menu:이미지[빌드]를 선택하십시오.

. 기본 `latest`이 아닌 다른 버전을 원하는 경우 다른 태그 이름을 추가하십시오(컨테이너에만 해당).

. `Build Profile` 및 `Build Host`을(를) 선택하십시오.

+

[NOTE]
====
빌드 필드 오른쪽에 있는 `Profile Summary`을(를) 확인하십시오.
빌드 프로파일을 선택하면 선택한 프로파일에 대한 자세한 정보가 이 영역에 표시됩니다.
====

. 빌드를 예약하려면 btn:[빌드] 버튼을 클릭하십시오.

_____



[[at.images.docker.importing]]
=== 이미지 임포트

이미지를 얻는 두 번째 방법은 임의의 이미지를 가져오고 검사하는 것입니다.
이를 위해 왼쪽 탐색 바에서 menu:이미지[이미지 목록]을 선택하십시오.
`Import` 대화 상자의 텍스트 상자를 작성하십시오.
처리되면 가져온 이미지는 `Image List` 페이지에 나열됩니다.

.절차 이미지 가져오기
[role=procedure]
_____

. menu:이미지[이미지 목록]에서 btn:[임포트]를 클릭하여 `Import Image` 대화 상자를 엽니다.

. `Import Image` 대화 상자에서 다음 필드를 작성하십시오:

+

이미지 저장소:::
검사를 위해 이미지를 끌어오는 레지스트리입니다.

이미지 이름:::
레지스트리에 있는 이미지의 이름입니다.

이미지 버전:::
레지스트리에 있는 이미지의 버전입니다.

빌드 호스트:::
이미지를 끌어오고 검사하는 빌드 호스트입니다.

활성화 키:::
이미지를 검사하는 소프트웨어 채널에 대한 경로를 제공하는 활성화 키입니다.

. 확인을 위해 btn:[임포트]를 클릭하십시오.

_____

이미지 항목이 데이터베이스에 생성되고 `Inspect Image`에 대한 {productname} 작업이 예약됩니다.

처리되면 가져온 이미지를 `Image List`에서 찾을 수 있습니다.
`Build` 열에서 이미지를 가져왔음을 나타내기 위해 다른 아이콘이 있습니다.

== OS 이미지

OS 이미지는 Kiwi 빌드 시스템에 의해 생성됩니다.
출력 이미지는 사용자 정의가 가능하며 PXE 부팅, QCOW2, LiveCD 또는 기타 유형의 이미지일 수 있습니다.

Kiwi 빌드 시스템에 대한 자세한 정보는 https://osinside.github.io/kiwi/[Kiwi 문서]를 참조하십시오.



[[at.images.kiwi.requirements]]
=== 요구사항

Kiwi 이미지 빌드 기능은 {salt} 클라이언트에서 {sles}{nbsp}15 및 {sles}{nbsp}12를 실행하는 경우 사용할 수 있습니다.
//SLE15 images support is not yet released for MLM4, but is part of MLM4.0.4 as tech preview
//From {sles}{nbsp}15, `kiwi-ng` is used instead of the legacy Kiwi.

Kiwi 이미지 구성 파일 및 구성 스크립트는 다음 위치 중 한 곳에서 액세스할 수 있어야 합니다.

* Git 리포지토리
* HTTP 또는 HTTPS 호스팅 tar 아카이브
* 빌드 호스트 상의 로컬 디렉토리

git로 제공되는 완전한 Kiwi 리포지토리의 예는 https://github.com/SUSE/manager-build-profiles/tree/master/OSImage.을 참조하십시오.

[NOTE]
====
Kiwi로 빌드된 OS 이미지를 실행하는 호스트에는 최소 1{nbsp}GB의 RAM이 필요합니다.
디스크 공간은 이미지의 실제 크기에 따라 다릅니다.
자세한 내용은 기본 시스템의 문서를 참조하십시오.
====



=== 이미지를 빌드할 때 HTTP/HTTPS 프록시를 통해 Git 리포지토리에 접근하기

빌드 호스트가 HTTP/HTTPS 프록시를 통해서만 접근할 수 있는 git 저장소에서 소스를 가져와야 할 때, {salt} 상태에 의해 호출된 git 클라이언트가 항상 시스템 전체 프록시 설정(예: `/etc/sysconfig/proxy` 또는 환경 변수)을 인식하지 않기 때문에 빌드 중에 git 타임아웃이 발생할 수 있습니다.

비대화형 빌드를 위해 git이 프록시를 사용하도록 하려면 `/etc/gitconfig`를 http.proxy 항목과 함께 생성하십시오:

[source,console]
----
# /etc/gitconfig
[http]
    proxy = http://proxy.example.com:3128
# or for HTTPS
[http]
    proxy = https://proxy.example.com:3128
----

프록시가 인증을 요구하는 경우, URL에 자격 증명을 포함하는 대신 자격 증명 도우미를 선호하거나 자격 증명 저장소를 사용하십시오.



==== {salt}으로 자동화하기

구성 관리와 같은 방식으로 빌드 호스트에서 `/etc/gitconfig`을 관리할 수 있습니다.
{salt} 상태 파일을 생성하고 구성 채널을 통해 빌드 호스트에 할당한 다음, {productname} UI에서 고상태를 적용하십시오.

시스템 사용자 정의 정보 필드에서 프록시 주소를 가져오려면 `custom_info:` 필러 접두사를 사용하십시오:

[source,yaml]
----
/etc/gitconfig:
  file.managed:
    - user: root
    - group: root
    - mode: '0644'
    - contents: |
        [http]
            proxy = {{ salt['pillar.get']('custom_info:build_server:git_proxy', 'http://proxy.example.com:3128') }}
----

=== Kiwi 기반 이미지 임포트는 지원되지 않습니다.

기존의 레거시 Kiwi 및 KiwiNG 도구 외에도, {productname}에 컨테이너화된 Kiwi 이미지 빌드 시스템이 도입되었습니다.


==== 구성 채널 관리 

관리자는 다음 필러 또는 사용자 정의 값을 사용하여 기본 동작을 재정의할 수 있습니다.
이들을 구성하려면 {webui}의 \menu:[시스템>사용자 정의 시스템 정보]로 이동하여 필요한 키를 생성합니다.

사용되는 빌드 시스템은 기본 OS 또는 특정 필러 값에 따라 달라집니다.

* SLE 11 / SLE 12: 레거시 Kiwi v7
* SLE 15:  KiwiNG (v9 및 컨테이너화된 Kiwi 10)


관리자는 다음 필러 또는 사용자 정의 값을 사용하여 기본 동작을 재정의할 수 있습니다.

* `use_kiwi_ng`: Kiwi 9 사용 강제,
* `use_kiwi_container`: 컨테이너화된 Kiwi 10 사용 강제.
  이를 활성화하려면 값을 `1`로 설정합니다.
* `use_bundle_build`: 추가 KIWI 번들 빌드 아티팩트(PXE 파일이 포함된 `.install.tar` 및 `.raw.xz` 등)를 서버에 업로드합니다.
  이를 활성화하려면 값을 `true`(으)로 설정합니다.


==== 서버 구성

SLES 15 프로파일에는 Kiwi 9이 사용되는 반면, 기본 컨테이너 동작에는 Kiwi 10이 사용되므로 SUSE Linux Enterprise 15 프로파일용 컨테이너화된 빌드 호스트를 사용할 때는 특정 구성이 필요합니다.

SLES 15 프로파일에 올바른 버전이 사용되도록 하려면 `kiwi_image` 사용자 정의 정보 키를 다음 값으로 정의해야 합니다:

* `Key`: kiwi_image
* `Value`: `registry.suse.com/bci/kiwi:9`

이 키가 설정되지 않은 경우 시스템은 최신 버전(예: `registry.suse.com/bci/kiwi:10.2`)을 기본값으로 사용하며, 이로 인해 SLES 15 프로파일에서 빌드 문제가 발생할 수 있습니다.

모든 종류의 이미지를 {productname}로 빌드하려면 빌드 호스트를 생성하고 구성해야 합니다.
OS 이미지 빌드 호스트는 {salt} 클라이언트가 {sles}{nbsp}15 (SP2 이상) 또는 {sles}{nbsp}12 (SP4 이상)에서 실행되고 있습니다.

예를 들어, Raspberry PI {sles}{nbsp}15 SP3 이미지는 {sles}{nbsp}15 SP3을 실행하는 Raspberry PI (aarch64 아키텍처) 빌드 호스트에서 빌드해야 합니다.

. menu:시스템 세부 정보[속성] 탭으로 이동하여 `Add-on System Type` > `OS Image Build Host` 상자를 선택합니다. 

. btn:[속성 업데이트]로 확인합니다.

. menu:시스템 세부 사항[소프트웨어> 소프트웨어 채널]로 이동하고 빌드 호스트 버전에 따라 필요한 소프트웨어 채널을 활성화합니다.

+

* {sles}{nbsp}12 빌드 호스트에는 {productname} 클라이언트 도구(`SLE-Manager-Tools12-Pool` 및 `SLE-Manager-Tools12-Updates`)가 필요합니다.

*  예약하고 btn:[확인]을 클릭합니다.

  또는 {productname} 서버 명령줄에서 고상태를 적용합니다:

+

----
salt '$your_client' state.highstate
----

==== {productname} 웹 서버 공인 인증서 RPM

빌드 호스트 프로비저닝은 {productname} 인증서 RPM을 빌드 호스트로 복사합니다.
이 인증서는 {productname}에서 제공하는 리포지토리에 접근하는 데 사용됩니다.

인증서는 `mgr-package-rpm-certificate-osimage` 패키지 스크립트에 의해 RPM으로 패키징됩니다.
패키지 스크립트는 새로운 {productname} 설치 중에 자동으로 호출됩니다.

`spacewalk-certs-tools` 패키지를 업그레이드할 때, 업그레이드 시나리오는 기본값을 사용하여 패키지 스크립트를 호출합니다.
그러나 인증서 경로가 변경되었거나 사용할 수 없는 경우, 업그레이드 절차가 완료된 후 `--ca-cert-full-path <path_to_certificate>`를 사용하여 패키지 스크립트를 수동으로 호출합니다.


==== 패키지 스크립트 호출 예제

[source,bash]
----
/usr/sbin/mgr-package-rpm-certificate-osimage --ca-cert-full-path /root/ssl-build/RHN-ORG-TRUSTED-SSL-CERT
----

인증서가 포함된 RPM 패키지는 다음과 같이 Salt가 액세스할 수 있는 디렉토리에 저장됩니다.
----
/usr/share/susemanager/salt/images/rhn-org-trusted-ssl-cert-osimage-1.0-1.noarch.rpm
----

인증서가 포함된 RPM 패키지는 다음 로컬 빌드 호스트 리포지토리에 제공됩니다.
----
/var/lib/Kiwi/repo
----

[IMPORTANT]
====
빌드 소스에 {productname} SSL 인증서가 포함된 RPM 패키지를 지정하고, Kiwi 구성에 `rhn-org-trusted-ssl-cert-osimage`이(가) `bootstrap` 섹션의 필수 패키지로 포함되어 있는지 확인합니다.

.config.xml
[source,xml]
----
...
  <packages type="bootstrap">
    ...
    <package name="rhn-org-trusted-ssl-cert-osimage" bootinclude="true"/>
  </packages>
...
----
====

[[at.images.kiwi.sourceexample]]
==== Kiwi 소스의 예제


Kiwi 소스는 최소한 `config.xml`로 구성됩니다.
보통 `config.sh`와 `images.sh`도 함께 존재합니다.
소스에는 `root` 하위 디렉토리 아래에 최종 이미지에 설치할 파일도 포함될 수 있습니다.

Kiwi 빌드 시스템에 대한 정보는 https://osinside.github.io/kiwi/[Kiwi 문서]를 참조하십시오.

{suse}은 https://github.com/SUSE/manager-build-profiles[SUSE/manager-build-profiles] 공개 GitHub 리포지토리에서 완전한 기능을 갖춘 이미지 소스의 예를 제공합니다.

.JeOS config.xml의 예
// ###example-script-1###
[source, xml]
----
<?xml version="1.0" encoding="utf-8"?>

<image schemaversion="6.1" name="POS_Image_JeOS6">
    <description type="system">
        <author>Admin User</author>
        <contact>noemail@example.com</contact>
        <specification>SUSE Linux Enterprise 12 SP3 JeOS</specification>
    </description>
    <preferences>
        <version>6.0.0</version>
        <packagemanager>zypper</packagemanager>
        <bootsplash-theme>SLE</bootsplash-theme>
        <bootloader-theme>SLE</bootloader-theme>

        <locale>en_US</locale>
        <keytable>us.map.gz</keytable>
        <timezone>Europe/Berlin</timezone>
        <hwclock>utc</hwclock>

        <rpm-excludedocs>true</rpm-excludedocs>
        <type boot="saltboot/suse-SLES12" bootloader="grub2" checkprebuilt="true" compressed="false" filesystem="ext3" fsmountoptions="acl" fsnocheck="true" image="pxe" kernelcmdline="quiet"></type>
    </preferences>
    <!--    CUSTOM REPOSITORY
    <repository type="rpm-dir">
      <source path="this://repo"/>
    </repository>
    -->
    <packages type="image">
        <package name="patterns-sles-Minimal"/>
        <package name="aaa_base-extras"/> <!-- wouldn't be SUSE without that ;-) -->
        <package name="kernel-default"/>
        <package name="venv-salt-minion"/>
        ...
    </packages>
    <packages type="bootstrap">
        ...
        <package name="sles-release"/>
        <!-- this certificate package is required to access {productname} repositories
             and is provided by {productname} automatically -->
        <package name="rhn-org-trusted-ssl-cert-osimage" bootinclude="true"/>

    </packages>
    <packages type="delete">
        <package name="mtools"/>
        <package name="initviocons"/>
        ...
    </packages>
</image>
----



[[at.images.kiwi.building]]
=== 이미지 빌드

{webui}을 사용하여 이미지를 빌드하거나 가져오는 방법은 두 가지가 있습니다.
menu:이미지[빌드]를 선택하거나 menu:이미지[프로파일] 목록에서 빌드 아이콘을 클릭하십시오.

.절차 이미지를 빌드하는 중입니다.
[role=procedure]
_____

. menu:이미지[빌드]를 선택하십시오.

. 기본 `latest`이(가) 아닌 다른 버전을 원하는 경우 다른 태그 이름을 추가합니다(컨테이너에만 적용됨).

. `Image Profile` 및 `Build Host`을(를) 선택합니다.

+

[NOTE]
====
`Profile Summary`이(가) 빌드 필드 오른쪽에 표시됩니다.
빌드 프로파일을 선택하면 선택한 프로파일에 대한 자세한 정보가 여기에 표시됩니다.
====

+

. 빌드를 예약하려면 btn:[빌드] 버튼을 클릭하십시오.

_____


[IMPORTANT]
====
빌드 서버는 이미지 빌드 프로세스 중에 어떤 형태의 자동 마운팅도 실행할 수 없습니다.
해당되는 경우, Gnome 세션이 루트 사용자로 실행되지 않도록 하십시오.
자동 마운팅이 실행 중인 경우, 이미지 빌드는 성공적으로 완료되지만 이미지의 체크섬이 다르며, 이로 인해 오류가 발생합니다.
====


[[at.images.kiwi.inspection]]

이미지가 성공적으로 빌드된 후, 검사 단계가 시작됩니다.
검사 단계 동안 {susemgr}은 이미지에 대한 정보를 수집합니다:

* 이미지에 설치된 패키지 목록
* 이미지의 체크섬
* 이미지 유형 및 기타 이미지 세부 사항

[NOTE]
====
빌드된 이미지 유형이 `PXE`인 경우, {salt} 필라도 생성됩니다.
이미지 필라는 데이터베이스에 저장되며 {salt} 서브시스템은 생성된 이미지에 대한 세부정보에 접근할 수 있습니다.
세부정보에는 이미지 파일이 위치한 곳, 제공된 이미지 체크섬, 네트워크 부팅에 필요한 정보 등이 포함됩니다.

생성된 열은 연결된 모든 클라이언트에서 사용할 수 있습니다.
====



[[at.images.kiwi.troubleshooting]]
=== 문제 해결

이미지를 빌드하려면 여러 종속 단계가 필요합니다.
빌드가 오류가 발생할 때, {salt} 상태 결과와 빌드 로그를 조사하면 오류의 원인을 파악하는 데 도움이 될 수 있습니다.
빌드가 오류가 발생할 때 이러한 점검을 수행할 수 있습니다:

* 빌드 호스트는 빌드 소스에 액세스할 수 있습니다.
* 빌드 호스트와 {productname} 서버 모두에 이미지를 위한 충분한 디스크 공간이 있습니다.
* 활성화 키에 연결된 올바른 채널이 있습니다.
* 사용된 빌드 소스가 유효합니다.
* {productname} 공개 인증서가 포함된 RPM 패키지가 최신 상태이며 `/usr/share/susemanager/salt/images/rhn-org-trusted-ssl-cert-osimage-1.0-1.noarch.rpm`에서 사용할 수 있습니다.
  공개 인증서 RPM을 새로 고치는 방법에 대한 자세한 내용은 <<at.images.kiwi.buildhost>>을 참조하십시오.



[[at.images.kiwi.limitations]]
=== 제한 사항

이 섹션에는 이미지 관련 작업을 할 때 알려진 몇 가지 문제가 포함되어 있습니다.

* HTTP 소스 또는 git 리포지토리에 액세스하기 위해 사용되는 HTTPS 인증서는 사용자 정의 상태 파일을 통해 클라이언트에 배포하거나 수동으로 구성해야 합니다.
* Kiwi 기반 이미지 임포트는 지원되지 않습니다.



[[at.images.listing]]
== 빌드 이미지 목록

사용 가능한 빌드 이미지를 나열하려면 menu:이미지[이미지 목록]을 선택하십시오.
모든 이미지의 목록이 표시됩니다.

이미지에 대해 표시되는 데이터에는 이미지 `Name`, `Version`, `Revision` 및 빌드 `Status`이(가) 포함됩니다.
이미지에 대한 가능한 패치 및 패키지 업데이트 목록과 함께 이미지 업데이트 상태를 확인할 수 있습니다.

OS 이미지의 경우 `Name` 및 `Version` 필드는 Kiwi 소스에서 유래하며 성공적인 빌드가 끝날 때 업데이트됩니다.
빌드 중 또는 오류가 발생한 빌드 후에 이 필드는 프로필 이름을 기반으로 한 임시 이름을 표시합니다.

`Revision`은(는) 각 성공적인 빌드 후에 자동으로 증가합니다. OS 이미지의 경우, 여러 수정이 저장소에 공존할 수 있습니다.

컨테이너 이미지의 경우, 저장소는 최신 수정만 보유합니다.
이전 수정(패키지, 패치 등)에 대한 정보는 보존되며, `Show obsolete` 체크박스를 사용하여 나열할 수 있습니다.

이미지에서 btn:[세부정보] 버튼을 클릭하면 자세한 보기를 제공합니다.
자세한 보기에는 관련 패치의 정확한 목록, 이미지 내에 설치된 모든 패키지 목록 및 빌드 로그가 포함됩니다.

btn:[삭제] 버튼을 클릭하면 목록에서 이미지가 삭제됩니다.
또한 관련 필라, OS 이미지 저장소의 파일 및 구식 수정이 삭제됩니다.

[NOTE]
====
빌드가 성공한 후 상태를 검사한 경우에만 패치 및 패키지 목록을 사용할 수 있습니다.
====
