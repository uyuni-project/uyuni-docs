가상 머신 관리를 위해 {productname}을(를) 사용 중인 경우, 가상 머신의 클론을 만드는 것이 유용할 수 있습니다.
클론은 기존 디스크의 정확한 복사본인 기본 디스크를 사용하는 가상 머신입니다.

VM을 복제하면 시간을 크게 절약할 수 있지만, 디스크에 중복된 식별 정보가 있으면 문제가 발생할 수 있습니다.

이미 등록된 클라이언트가 있는 경우, 해당 클라이언트의 클론을 만들고 클론을 등록하려고 하면, {productname}이(가) 두 개의 별도 클라이언트로 등록하기를 원할 것입니다.
그러나 원래 클라이언트와 클론의 머신 ID가 동일하면, {productname}는 두 클라이언트를 하나의 시스템으로 등록하고 기존 클라이언트 데이터는 클론의 데이터로 덮어씌워집니다.

이는 클론의 머신 ID를 변경하여 {productname}이(가) 이를 서로 다른 두 클라이언트로 인식하도록 하면 해결할 수 있습니다.

[NOTE]
====
이 절차의 각 단계는 클론 클라이언트에서 수행됩니다.
이 절차는 원래 클라이언트를 조작하지 않으며, 원래 클라이언트는 {productname}에 등록된 상태로 유지됩니다.
====



.절차 클론된 Salt 클라이언트에서 중복 머신 ID 해결
[role=procedure]
____

. *초기 시스템 구성*


.. 복제된 머신에서 호스트 이름과 IP 주소를 변경합니다.
    `/etc/hosts`에 변경사항과 올바른 호스트 항목이 포함되어 있는지 반드시 확인하세요.


. *중복 머신 ID 해결*

.. _systemd를 지원하는 배포판의 경우:_

...   머신의 머신 ID가 동일한 경우, 복제된 각 클라이언트에서 루트 권한으로 파일을 삭제하고 다시 생성합니다.

+

----
rm /etc/machine-id
rm /var/lib/dbus/machine-id
rm /var/lib/zypp/AnonymousUniqueId
dbus-uuidgen --ensure
systemd-machine-id-setup
----

+

... 클론된 머신에도 `/var/log/journal/`에 폴더가 있다면, 새 머신 ID에 맞게 이름을 변경해야 합니다. 이름이 일치하지 않으면 `journalctl`에서 로그를 가져올 수 없고 `podman logs`에는 아무것도 표시되지 않습니다.

+

----
mv /var/log/journal/* /var/log/journal/$(cat /etc/machine-id)
----

+

.. _systemd를 지원하지 않는 배포판의 경우:_ 

...  루트 권한으로 dbus에서 머신 ID를 생성합니다.

+

----
rm /var/lib/dbus/machine-id
rm /var/lib/zypp/AnonymousUniqueId
dbus-uuidgen --ensure
----

+ 

[IMPORTANT]
====

* 나중에 {rhel}로 해방될 {sll} 8.10 서버를 복제하는 경우, 커널 구성 파일을 수정하기 위한 추가 단계를 수행해야 합니다. 


* {rhel}은 머신 ID를 사용하여 `/boot/loader/entries`에 커널 항목을 생성합니다. 
이 단계를 수행하지 않으면 해방 후 오래된 커널 항목과 새로운 커널 항목이 혼합되어 나타나며, {sll} 커널은 오래된 항목을 대체하는 대신 새로운 항목을 생성합니다.


* 머신 ID 변경 후 전환하기 전에 다음 명령을 실행하십시오.

+

----
sudo rm -rf /boot/loader/entries/
sudo for ver in $(rpm -q kernel --qf '%{VERSION}-%{RELEASE}.%{ARCH}\n'); do echo "Reinstalling kernel $ver..."; sudo kernel-install add $ver /lib/modules/$ver; done
sudo grub2-mkconfig -o /boot/efi/EFI/redhat/grub.cfg
----

* {rhel} 8.10 서버를 해방하는 방법에 대한 자세한 정보와 예시는 xref:common-workflows:workflow-liberate-rhel-with-secureboot.adoc[]를 참조하십시오.

====

. *클라이언트{salt} 재구성*

.. 클라이언트에 여전히 동일한 Salt 클라이언트 ID가 있는 경우, 각 클라이언트에서 `minion_id` 파일을 삭제하세요(클라이언트를 재시작하면 재생성될 때 FQDN이 사용됩니다).

...  Salt 번들 클라이언트의 경우 다음과 같습니다.

+

----
rm /etc/salt/minion_id
rm -rf /etc/salt/pki
----

+

... Salt 번들 클라이언트의 경우 다음과 같습니다.

+

----
rm /etc/venv-salt-minion/minion_id
rm -rf /etc/venv-salt-minion/pki
----

+

.. 온보딩 페이지에서 승인된 키를 삭제하고 {productname}에서 시스템 프로파일을 삭제한 후, 클라이언트를 재시작하십시오.

... Salt 번들 클라이언트의 경우 다음과 같습니다.

+

----
service salt-minion restart
----

+

... Salt 번들 클라이언트의 경우 다음과 같습니다. 


+

----
service venv-salt-minion restart
----

+

.. 클라이언트를 다시 등록합니다.
   이제 각 클라이언트는 서로 다른 `/etc/machine-id`을(를) 가지며 `System Overview` 페이지에 올바르게 표시되어야 합니다.
