[[tshoot-wicked-to-networkmanager]]
= 호스트 OS 업그레이드 후 wicked를 NetworkManager로 교체
:revdate: 2026-06-08
:page-revdate: {revdate}

////
Applies to: SUSE Multi-Linux Manager 5.0 / 5.1 on SL Micro 6.1
Platform: Microsoft Azure (cloud image deployments)

Cause: After upgrading the host OS from SLE Micro 5.5 to SL Micro 6.1, wicked remains active.
Consequence: Wicked packages become orphaned with no repository or security updates.
Fix: Install NetworkManager inside a transactional shell, remove wicked, enable NetworkManager, and reboot.
Result: NetworkManager manages the network and the system is in a supported configuration.
////


== 문제 설명

일부 경우, {productname} 호스트 OS를 SLE Micro 5.5에서 SL Micro 6.1로 업그레이드한 후, 시스템이 활성 네트워크 관리자로 wicked를 계속 실행할 수 있습니다.

이것은 지원되지 않는 구성입니다.

이전 설치에서 가져온 wicked 패키지는 SL Micro 6.1 리포지토리에서 더 이상 사용할 수 없으므로 고아가 되며 보안 업데이트를 받지 않습니다.
 
[NOTE]
====
{productname}은 SL Micro 호스트에서 컨테이너로 실행됩니다.
네트워크 관리는 호스트 OS에 의해 완전히 처리됩니다.
이 절차 동안 {productname} 컨테이너는 변경이 필요하지 않습니다.
====
 
다음 증상은 시스템이 영향을 받았음을 확인합니다:
 
* `systemctl status wicked`은(는) `active (exited)`을(를) 표시합니다
* `systemctl status NetworkManager`은(는) `Unit NetworkManager.service could not be found`을(를) 표시합니다
* `systemctl show -p Id network.service`은(는) `Id=wicked.service`을(를) 반환합니다
* `sudo zypper se -s wicked`은(는) 리포지토리 없이 `(System Packages)` 아래에 나열된 wicked를 표시합니다
 
[WARNING]
====
이 절차는 {productname} 클라우드 이미지의 표준 구성인 DHCP를 사용하는 시스템에 적용됩니다.
배포가 정적 IP 주소, VLAN, 사용자 정의 본딩 또는 기타 비표준 네트워크 구성을 사용하는 경우 진행하지 마십시오.
구성에 대한 구체적인 안내는 SUSE 지원팀에 문의하십시오.
====

== 전제 조건
 
시작하기 전에:
 
* 시스템에 대한 아웃 오브 밴드 콘솔 접근이 있는지 확인하십시오(예: 클라우드 플랫폼이나 하이퍼바이저에서 제공하는 직렬 콘솔).
재부팅 후 네트워크가 올바르게 복구되지 않으면 이것이 복구 경로입니다.
* {productname} 컨테이너가 실행 중이고 정상인지 확인하십시오:

+

----
sudo podman ps
----

* 호스트의 현재 IP 주소와 라우팅 테이블을 기록하십시오.
마이그레이션 후 변경되지 않았는지 확인할 것입니다:

+

----
ip addr show eth0
ip route show
----
 
 
== 마이그레이션 절차

.절차 wicked를 NetworkManager로 마이그레이션
[role=procedure]
____

. 변경하기 전에 스냅샷을 찍으십시오.
  문제가 발생하면 이것이 롤백 지점입니다.

+

----
sudo snapper create -d "Pre-NetworkManager-Migration"
----

+ 

출력에서 스냅샷 번호를 기록하십시오.
롤백할 경우 필요합니다.

+ 
----
sudo snapper list
----

. 변경 사항을 적용하기 위해 트랜잭션 업데이트 셸을 엽니다.

+

[IMPORTANT]
====
여기에서 이루어진 모든 변경 사항은 원자적으로 스테이징되며, 다음 단계에서 재부팅 후에만 적용됩니다.
====

+

----
sudo transactional-update shell
----

+
 
셸 안에서 NetworkManager를 설치합니다:
 
+

----
zypper in NetworkManager
----

+
 
프롬프트가 표시되면 `y`로 확인합니다.
NetworkManager와 그 의존성은 SL Micro-6.1 리포지토리에서 설치됩니다.

+
 
고아의 wicked 패키지를 제거합니다:

+
 
----
zypper rm wicked wicked-service cockpit-wicked
----

+
 
프롬프트가 표시되면 `y`로 확인합니다.

+
 
[NOTE]
====
wicked 제거 중 다음과 유사한 경고가 표시됩니다:
 
 warning: file wicked: remove failed: No such file or directory
 
이 경고는 예상되는 것입니다.
OS 업그레이드 후 일부 wicked 파일이 이미 없었습니다.
제거가 성공적으로 완료됩니다.
중요한 심볼릭 링크가 제거되었는지 확인하려면 다음 줄을 찾습니다:
 
 Removed "/etc/systemd/system/network.service".
 Removed "/etc/systemd/system/multi-user.target.wants/wicked.service".
====

+
 
NetworkManager를 활성화하여 다음 부팅 시 시작되도록 합니다:

+ 

----
systemctl enable NetworkManager
----

+
 
다음 심볼릭 링크가 생성된 것을 확인해야 합니다:
 
 Created symlink /etc/systemd/system/network.service → /usr/lib/systemd/system/NetworkManager.service.
 Created symlink /etc/systemd/system/multi-user.target.wants/NetworkManager.service → ...

+
 
트랜잭션 셸을 종료합니다.

+
 
----
exit
----

 
. 재부팅합니다.

+
 
[WARNING]
====
시스템은 재부팅 중에 네트워크 연결을 잃게 됩니다.
SSH 세션이 종료됩니다.
필요한 경우 아웃 오브 밴드 콘솔을 사용하여 재부팅을 모니터링합니다.
====

+

----
sudo reboot
----

 
. 시스템이 다시 시작된 후 마이그레이션을 확인합니다.
  호스트에서 다음 검사를 실행합니다.

+
 
`network.service`이(가) 이제 NetworkManager를 가리키는지 확인합니다:

+
 
----
systemctl show -p Id network.service
----

+
 
예상 출력:
 
 Id=NetworkManager.service

+
 
wicked 패키지가 완전히 제거되었는지 확인하십시오:

+
 
----
rpm -q wicked wicked-service
----

+
 
예상 출력:
 
 package wicked is not installed
 package wicked-service is not installed

+
 
IP 주소가 변경되지 않았는지 확인하십시오:

+
 
----
ip addr show eth0
----

+
 
IP 주소는 이전 마이그레이션에서 기록된 것과 일치해야 합니다.
변경된 경우, 미니언은 {productname}에 대한 연결을 잃게 됩니다.
// For more information, see xref:tshoot-wicked-to-networkmanager.adoc#ts-ip-changed[IP Address Changed After Reboot].

+
 
라우팅 테이블이 온전한지 확인하십시오:

+
 
----
ip route show
----

+
 
출력을 이전 마이그레이션에서 기록된 라우팅 테이블과 비교하고 모든 경로가 존재하는지 확인하십시오.

+
 
[NOTE]
====
부팅 직후 경로가 누락된 경우, 90초를 기다린 후 다시 확인하십시오.
`cloud-netconfig` 서비스는 60초의 타이머로 실행되며, 누락된 클라우드 제공업체별 경로를 자동으로 복원합니다.
====

+
 
{productname} 컨테이너가 실행 중인지 확인하십시오:

+
 
----
sudo podman ps
----

+
 
컨테이너는 부팅 후 몇 분 이내에 상태 `healthy`를 표시해야 합니다.

____

 
 
== 마이그레이션 후의 변경 사항
 
[cols="1,2", options="header"]
|===
| 항목 | 알림
 
| IP 주소
| DHCP를 사용할 때 변경되지 않음.
NetworkManager는 DHCP를 통해 동일한 주소를 획득합니다.
 
| 슬레이브 또는 본딩 인터페이스
| 슬레이브 인터페이스는 존재하며 마스터에 연결되어 있습니다.
NetworkManager가 슬레이브 인터페이스를 거기에서 표시하지 않기 때문에 더 이상 `ip addr show`에 나타나지 않습니다.
`ip link show <interface>`을(를) 사용하여 해당 항목이 있는지 확인합니다.
이는 예상되는 동작이며 네트워크 성능에 영향을 미치지 않습니다.
 
| 클라우드 제공업체별 경로
| `cloud-netconfig`에 의해 관리되며, 이는 NetworkManager와 독립적으로 동작하며 스위치 후에도 계속 운영됩니다.
부팅 후 경로가 누락된 경우 60초 이내에 복원됩니다.
 
| IPv6 링크 로컬 주소
| 기본 인터페이스의 링크 로컬 주소는 NetworkManager가 wicked와 다르게 생성하기 때문에 변경됩니다.
이는 IPv4를 사용하는 표준 {productname} 배포에 영향을 미치지 않습니다.
 
| 기본 경로 메트릭
| NetworkManager는 기본 경로에 명시적인 `metric 100`을(를) 추가합니다.
이는 올바른 동작이며 부정적인 영향을 미치지 않습니다.
 
| NetworkManager 연결 파일
| NetworkManager는 tmpfs 위치인 `/var/run/NetworkManager/system-connections/`에 연결 정의를 자동으로 생성합니다.
이들은 DHCP에서 매 부팅 시 재생성됩니다.
원래 wicked 구성은 `/etc/sysconfig/network/ifcfg-*`에 남아 있지만 NetworkManager에 의해 읽히지 않습니다.
 
| {productname} 컨테이너
| 적용되지 않음.
수정이나 재시작 없이 계속 실행됩니다.
|===
 
 
== 롤백
 
시스템이 재부팅 후 올바르게 돌아오지 않으면 1단계에서 찍은 스냅샷을 사용하여 롤백하십시오.
 

=== 부트 메뉴를 통해 롤백

.절차 부트 메뉴를 통해 롤백 중
[role=procedure]
____
. 아웃 오브 밴드 콘솔을 사용하여 시스템에 연결하십시오.

. 부팅 중 GRUB 메뉴가 나타나면 1단계에서 찍은 스냅샷을 선택하십시오.

. 시스템은 wicked가 활성화된 상태로 사전 마이그레이션 상태로 부팅됩니다.
____

 
=== 명령줄을 통해 롤백
 
시스템이 부팅되지만 네트워크가 제대로 작동하지 않는 경우:
 
----
sudo transactional-update rollback <snapshot-number>
sudo reboot
----
 
[NOTE]
====
롤백 후 시스템이 고아가 된 wicked 상태로 돌아갑니다.
시스템은 작동하지만 wicked는 지원되지 않습니다.
마이그레이션이 계속 실패하는 경우 SUSE 지원팀에 문의하십시오.
====
