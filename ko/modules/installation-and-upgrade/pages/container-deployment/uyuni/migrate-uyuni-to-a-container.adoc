레거시 {productname} 서버를 컨테이너로 마이그레이션하려면 새 머신이 필요합니다.

이 마이그레이션의 맥락에서 레거시 {productname} 서버(RPM 설치)는 때때로 _old server_라고도 불립니다.


== 요구 사항 및 고려 사항



=== 호스트 이름

인플레이스 마이그레이션은 불가능할 뿐만 아니라, 현재 마이그레이션 절차는 호스트 이름 변경 기능을 허용하지 않습니다.

따라서 새 서버의 FQDN(정규화된 도메인 이름)은 레거시 서버의 FQDN과 동일하게 유지됩니다.

[IMPORTANT]
====
마이그레이션 후에는 새 서버를 가리키도록 DHCP 및 DNS 레코드를 수동으로 업데이트해야 합니다.

자세한 내용은 <<migration-finalization>>을 참조하십시오.
====


== GPG 키

* 자체 신뢰 GPG 키는 마이그레이션되지 않습니다.
* RPM 데이터베이스에서만 신뢰되는 GPG 키는 마이그레이션되지 않습니다.
  따라서 `spacewalk-repo-sync`로 채널을 동기화하는 데 실패할 수 있습니다.
* 관리자는 실제 서버 마이그레이션을 수행한 후 이러한 키를 레거시 {productname} 설치에서 컨테이너 호스트로 수동으로 마이그레이션해야 합니다.


.절차 새 서버로 GPG 키를 수동으로 마이그레이션
. 4.3 서버의 키를 새 서버의 컨테이너 호스트로 복사합니다.
. 그 후, 마이그레이션된 서버에 각 키를 추가하려면 `mgradm gpg add <PATH_TO_KEY_FILE>` 명령을 사용하십시오.




=== 소스 서버에서의 초기 준비

마이그레이션은 복제해야 할 데이터의 양에 따라 매우 오랜 시간이 걸릴 수 있습니다.
다운타임을 줄이기 위해 레거시 서버의 모든 서비스가 계속 가동되는 동안 _초기 복제_, _재복제_ 또는 _최종 복제 및 전환_ 단계로 마이그레이션을 여러 번 실행할 수 있습니다.

기존 서버의 프로세스는 최종 마이그레이션 중에만 중지해야 합니다.

최종 복제가 아닌 모든 복제에 대해, 레거시 서버에서 서비스가 자동으로 중지되는 것을 방지하려면 `--prepare` 매개변수를 추가하십시오.
예:

[source,shell]
----
mgradm migrate podman <oldserver.fqdn> --prepare
----


.절차 소스 서버에서의 초기 준비

. {productname} 서비스 중지:
+
----
spacewalk-service stop
----
. PostgreSQL 서비스를 중지합니다.
+
----
systemctl stop postgresql
----


=== SSH 연결 준비

.절차 SSH 연결 준비
. 새 `root` 서버에 {productnumber}에 대한 SSH 키가 존재하는지 확인하십시오.
  키가 존재하지 않으면 다음 명령으로 생성합니다:
+
----
ssh-keygen -t rsa
----
. 4.3 서버에 비밀번호 없이 연결하려면 새 서버에서 SSH 구성 및 에이전트가 준비되어 있어야 합니다.
+
----
eval $(ssh-agent); ssh-add
----
+
[NOTE]
====
비밀번호 입력을 요구하지 않고 연결을 설정하기 위해 마이그레이션 스크립트는 새 서버에서 실행 중인 SSH 에이전트에 의존합니다.
에이전트가 아직 활성화되지 않았다면 `eval $(ssh-agent)`를 실행하여 시작하십시오.
그런 다음 `ssh-add`와(과) 개인 키 경로를 사용하여 실행 중인 에이전트에 SSH 키를 추가하십시오.
이 과정에서 개인 키의 비밀번호를 입력하라는 메시지가 표시됩니다.
====

. `ssh-copy-id`를 사용하여 공용 SSH 키를 레거시 {productname} 서버(`<oldserver.fqdn>`)에 복사하십시오.
  레거시 서버의 FQDN으로 `<oldserver.fqdn>`을(를) 교체하십시오:
+
----
ssh-copy-id <oldserver.fqdn>
----
+
SSH 키는 레거시 서버의 `~/.ssh/authorized_keys` 파일에 복사됩니다.
자세한 내용은 `ssh-copy-id` 매뉴얼 페이지를 참조하십시오.
. 새 서버에서 레거시 {productname} 서버로 SSH 연결을 설정하여 비밀번호가 필요하지 않은지 확인합니다.
  호스트 지문에 문제가 없어야 합니다.
  문제가 발생할 경우 `~/.ssh/known_hosts` 파일에서 오래된 지문을 제거하십시오.
  그런 다음 다시 시도하십시오.
  지문은 로컬 `~/.ssh/known_hosts` 파일에 저장됩니다.



=== 마이그레이션 수행

레거시 {productname}에서 컨테이너화된 {productname}로 마이그레이션을 계획할 때는 대상 인스턴스가 레거시 설정의 사양을 충족하거나 초과하는지 확인해야 합니다.
여기에는 메모리(RAM), CPU 코어, 스토리지 및 네트워크 대역폭이 포함되지만, 이에 국한되지 않습니다.

==== 사용자 정의 영구 스토리지 구성

영구 스토리지를 구성하는 것은 선택 사항이지만, 이는 컨테이너의 디스크 가득 차는 문제를 피하는 유일한 방법입니다.
`mgr-storage-server` 도구를 사용하여 사용자 지정 영구 스토리지를 구성하는 것이 강력히 권장됩니다.

자세한 내용은 `mgr-storage-server --help`을 참조하십시오.
자세한 내용은 [command]\``mgr-storage-server \--help``를 참조하십시오.  이 도구는 컨테이너 스토리지 및 데이터베이스 볼륨 생성을 간소화합니다.

다음 방법으로 명령 사용:

[source,shell]
----
mgr-storage-server <storage-disk-device> [<database-disk-device>]
----

[NOTE]
====
장치에는 파일 시스템이 없어야 합니다.
저장 장치에 파일 시스템이 존재하면 명령이 중단됩니다.
====

예:

[source,shell]
----
mgr-storage-server /dev/nvme1n1 /dev/nvme2n1
----

[NOTE]
====
이 명령은 `/var/lib/containers/storage/volumes`에 영구 스토리지 볼륨을 생성합니다.

자세한 내용은 다음을 참조하십시오.

* xref:installation-and-upgrade:container-management/persistent-container-volumes.adoc[]
* xref:administration:troubleshooting/tshoot-container-full-disk.adoc[]
====


==== 마이그레이션 수행


. 새 {productname} 서버를 설치하려면 다음 명령을 실행하십시오.
  레거시 서버의 FQDN으로 `<oldserver.fqdn>`을(를) 교체하십시오:
+
----
mgradm migrate podman <oldserver.fqdn>
----
. 신뢰할 수 있는 SSL CA 인증서를 마이그레이션합니다.


==== 인증서 마이그레이션

RPM의 일부로 설치되어 레거시 {productname}의 `/usr/share/pki/trust/anchors/` 디렉토리에 저장된 신뢰할 수 있는 SSL CA 인증서는 마이그레이션되지 않습니다.
{suse}는 컨테이너에 RPM 패키지를 설치하지 않기 때문에, 관리자는 마이그레이션 후 이전 설치에서 이러한 인증서 파일을 수동으로 마이그레이션해야 합니다:

.절차 인증서 마이그레이션
. 컨테이너 호스트에서 \``root``로 로그인합니다.
   예를 들어, `/local/ca.file`와(과) 같이 합니다.
. 다음을 사용하여 파일을 컨테이너에 복사합니다.
+
----
mgrctl cp /local/ca.file server:/etc/pki/trust/anchors/
----

==== zypper migration

[IMPORTANT]
====
`mgradm migrate` 명령을 성공적으로 실행한 후에도 모든 클라이언트의 {salt} 설정은 계속해서 레거시 서버를 가리킵니다.

새 {productnumber} 서버로 리디렉션하려면 인프라 수준(DHCP 및 DNS)에서 새 서버의 이름을 이전 서버와 동일한 FQDN 및 IP 주소를 사용하도록 변경해야 합니다.
====


[IMPORTANT]
====
마이그레이션에 문제가 발생하면 이전 시스템을 재시작할 수 있습니다.
루트로서 다음 명령을 사용하여 {postgresql} 및 스페이스워크 서비스를 재시작합니다:

[source,shell]
----
service postgresql start
spacewalk-service start
----

====

== 배포 준비

`mgradm migrate` 명령으로 마이그레이션을 실행하기 전에, 특히 마이그레이션 작업이 컨테이너를 처음부터 시작한다는 점을 고려하여 **Persistent Volumes**을 미리 정의하는 것이 필수적입니다.

자세한 내용은 이러한 볼륨을 준비하는 방법에 대한 설치 섹션을 xref:installation-and-upgrade:container-management/persistent-container-volumes.adoc[]에서 참조하십시오.



== 마이그레이션

다음 명령을 실행하여 새 {productname} 서버를 설치하고, **<oldserver.fqdn>**를 이전 서버의 적절한 FQDN으로 교체하십시오.

----
mgradm migrate podman <oldserver.fqdn>
----

또는

----
mgradm migrate kubernetes <oldserver.fqdn>
----

[IMPORTANT]
====

`mgradm migrate` 명령을 성공적으로 실행한 후에도 모든 클라이언트의 {salt} 설정은 계속해서 레거시 서버를 가리킵니다.
새 서버로 리디렉션하려면 인프라 수준(DHCP 및 DNS)에서 새 서버의 이름을 이전 서버와 동일한 FQDN 및 IP 주소를 사용하도록 변경해야 합니다.
====
