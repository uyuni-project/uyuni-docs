[[server-hardware-requirements]]
== 서버 요구사항


{sl-micro} {microversion} 또는 {sles} {bci-mlm} 중 하나가 컨테이너 호스트의 운영 체제입니다.

다음에서 {sles}이 설치된 호스트 운영 체제로 명시되는 경우는 중요할 때만 해당됩니다.
그렇지 않으면 {sl-micro} 또는 단순히 호스트 운영 체제라고 작성합니다.

운영 체제로 {sl-micro}을(를) 사용하는 컨테이너 호스트에는 디스크 여유 공간이 필요합니다:

* 기본 설치 시 최소 100GB
* 리포지토리 데이터용 공간 최소 130GB

선택한 소프트웨어의 양에 따라 운영 체제로 {sles}을(를) 사용하려면 상당히 많은 디스크 공간이 필요할 수 있습니다.

기본적으로 {productname} 서버 컨테이너는 미러링된 리포지토리(패키지 또는 제품), 데이터베이스 및 기타 데이터를 `/var/lib/containers/storage/volumes/` 디렉토리의 하위 디렉토리에 저장합니다.
이 디렉토리가 디스크 공간이 부족하면 리포지토리 동기화가 실패합니다.
미러링할 클라이언트 및 리포지토리의 수와 종류에 따라 `/var/lib/containers/storage/volumes/` 디렉토리에 필요한 공간을 추정하십시오.

파일 시스템 및 파티셔닝 세부정보에 대한 자세한 내용은 xref:installation-and-upgrade:hardware-requirements.adoc#install-hardware-requirements-storage[] 및 이 가이드의 설치 및 배포 섹션에 있는 자세한 설치 지침을 참조하십시오.

[[installation-swap-space]]
== 스왑 공간

작업 부하는 시스템마다 다릅니다.
무거운 또는 예측할 수 없는 작업 부하가 있는 시스템은 총 RAM에 관계없이 추가 스왑 공간의 이점을 누릴 수 있습니다.
가장 빠른 사용 가능한 저장소(예: SSD)에 스왑을 배치하는 것이 좋습니다. {suse}은 스왑 파티션 대신 스왑 파일을 사용하는 것을 권장합니다.

크기 권장 사항은 위 표를 참조하십시오.

다음 셸 명령 조각은 8GiB 스왑 파일을 생성합니다.

[NOTE]
====
BtrFS 파일 시스템상의 스왑 파일은 해당 볼륨의 스냅샷 생성을 방지합니다. `/var`은 다음 예시에서 이미 snapper 스냅샷에서 제외되어 있으므로, `/var/swap`을 안전하게 사용할 수 있습니다.
====

[source,shell]
----
## setup swapfile at /var/swap
# run following as a root user

# allocate 8GiB for swap file
fallocate -l 8G /var/swap
# ensure CoW is disabled for the swap file
chattr +C /var/swap
# allow only root access
chmod 600 /var/swap
# make swap file based on allocated file
mkswap /var/swap
# activate swap use for the running system
swapon /var/swap
# activate swap during the next boots
echo "/var/swap swap swap defaults 0 0" >> /etc/fstab
----




[[installation-postgresql-requirements]]
== 데이터베이스 요구사항

{postgresql}은 지원되는 유일한 데이터베이스입니다.
원격 {postgresql} 데이터베이스 또는 {postgresql} 데이터베이스와 함께 원격 파일 시스템(NFS 등)을 사용하는 것은 지원되지 않습니다.
다시 말해, {postgresql}는 {productname}를 위해 가장 빠른 저장 장치에 있어야 합니다.

[IMPORTANT]
====
잠재적인 성능 문제로 인해, {productname}에서 원격으로 {postgresql} 데이터베이스를 실행하는 것은 권장되지 않습니다.
이러한 환경이 가능하고 많은 경우 안정적이긴 하지만, 문제가 발생할 경우 데이터 손실의 위험이 항상 존재합니다.

ifeval::[{mlm-content} == true]
{suse}은 이러한 경우에 지원을 제공하지 못할 수 있습니다.
endif::[]

[[install-hardware-requirements-storage]]
== 영구 스토리지 및 권한

컨테이너를 배포할 때 기본적으로 영구 볼륨이 생성됩니다.

그러나 볼륨은 하나 이상의 별도의 저장 장치에 저장하는 것이 권장됩니다.
이러한 설정은 생산 환경에서 데이터 손실을 방지하는 데 도움이 됩니다.
컨테이너 배포 후에 이 작업을 수행할 수 있습니다.

저장 장치는 컨테이너를 처음 배포한 후에 설정하는 것이 가장 좋습니다.
자세한 내용은 xref:installation-and-upgrade:container-management/persistent-container-volumes.adoc[Persistent Container Volumes]을 참조하십시오.

모든 볼륨에 대해 XFS를 파일 시스템 유형으로 사용하는 것을 권장합니다.
리포지토리 저장용 디스크의 크기는 {productname}으로 관리할 배포판 및 채널의 수에 따라 다릅니다.
필요한 크기를 추정하는 가이드는 이 섹션의 표를 참조하십시오.

[IMPORTANT]
====
{cobbler} 또는 {postgresql} 저장소에 NFS를 사용하지 마십시오. SELinux 환경에서도 NFS를 사용하지 마십시오.
이러한 시나리오는 지원되지 않습니다.
====

{productname} 서버에서 이 명령을 사용하여 사용할 수 있는 모든 저장 장치를 찾습니다:

[source, shell]
----
hwinfo --disk | grep -E "Device File:"
----

`lsblk` 명령을 사용하여 각 장치의 이름 및 크기를 확인하십시오.

장치 이름과 함께 `mgr-storage-server` 명령을 사용하여 외부 디스크를 저장소의 위치로 설정하고, 선택적으로 자체 디스크에 데이터베이스의 위치로 설정하십시오:

[source, shell]
----
mgr-storage-server <storage-disk-device> [<database-disk-device>]
----

예:

[source, shell]
----
mgr-storage-server /dev/nvme1n1 /dev/nvme2n1
----

외부 스토리지 볼륨은 `/manager_storage` 및 `/pgsql_storage`에 마운트된 XFS 파티션으로 설정됩니다.

[NOTE]
====
이 명령은 `/var/lib/containers/storage/volumes`에 영구 스토리지 볼륨을 생성합니다.

자세한 내용은 xref:installation-and-upgrade:container-management/persistent-container-volumes.adoc[]을 참조하십시오.
====

채널 데이터와 데이터베이스에 동일한 저장 장치를 사용할 수 있습니다.
그러나 이는 권장되지 않으며, 채널 저장소가 증가하면 저장소가 가득 차 데이터베이스 무결성에 위험을 초래할 수 있습니다.
별도의 저장 장치를 사용하는 것이 성능을 향상시킬 수도 있습니다.
단일 저장 장치를 사용하려면 `mgr-storage-server`를 단일 장치 이름 파라미터와 함께 실행하십시오.

프록시를 설치하는 경우에는 `mgr-storage-proxy` 명령에서 1개의 장치 이름 파라미터만 사용할 수 있으며 외부 저장소 위치는 Squid 캐시로 설정됩니다.



== 논리 볼륨 관리(LVM)


// container-deployment/mlm/server-deployment-vm-mlm.adoc etc.
모든 종류의 가상 머신(VM)에 대해 LVM은 일반적으로 필요하지 않으며 권장되지 않습니다.
디스크 설정은 가상이며 볼륨을 위한 별도의 디스크가 가능하고 권장됩니다.

// container-deployment/mlm/server-deployment-mlm.adoc
다른 배포의 경우 볼륨에 대한 별도의 디스크도 권장됩니다.

{productname} 서버의 컨테이너 호스트에서 `mgr-storage-server` 명령은 `/var/lib/containers/storage/volumes` 디렉토리의 전체 내용을 별도의 디스크로 이동하고 `/var/lib/containers/storage/volumes`에 다시 마운트합니다.
