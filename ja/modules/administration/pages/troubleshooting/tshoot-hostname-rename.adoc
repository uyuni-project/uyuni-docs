{productname}サーバのホスト名をローカルで変更すると、{productname}のインストールが正常に動作しなくなります。
これは、データベースに変更が反映されていないためであり、クライアントやプロキシに変更が伝播するのを妨げます。



== サーバの名前変更

{productname} サーバのホスト名を変更する必要がある場合は、`mgradm server rename`コマンドを使用して変更できます。
このコマンドは、{postgresql}データベースの設定と{productname}の内部構造を更新します。



=== サーバ設定

コマンドは必須パラメータを取りませんが、コンテナホストのホスト名とは異なる場合は、新しいホスト名を受け取ることができます。

新しいホスト名に一致するSSL証明書を生成する必要がある場合は、SSL CAパスワードを提供する必要があります。
これは、設定ファイルを使用して安全に実現できます。

.プロシージャ:SSL CAパスワード用の設定ファイルを準備します。
[role=procedure]
____
. `config.yaml`ファイルを次のような内容で作成します。

+

[source,shell]
----
ssl:
  password: "<CA PASSWORD>"
----
____

このファイルは次のプロシージャで使用されます。

.プロシージャ:{productname}サーバの名前変更
[role=procedure]
____

. システムレベルのサーバのネットワーク設定を、DNSサーバでローカルおよびリモートで変更します。
    逆引き名前解決のための設定も提供する必要があります。
    ネットワーク設定の変更は、他のシステムの名前変更と同様に行われます。

. {productname}サーバを再起動して、新しいネットワーク構成を使用し、ホスト名が変更されていることを確認します。

. コンテナホストのコマンドラインから、次のコマンドを実行します。SSL CAパスワードを保存するためにファイルを作成した場合は、`-c config.yaml`を追加してください。

+

[source,shell]
----
mgradm server rename
----

+

新しいホスト名が解決できない場合、コマンドは失敗します。

____

名前変更手順は、サーバコンテナの再起動中にも行われます。
ログは、このコマンドを実行することで見つけることができます:

[source,shell]
----
mgrctl exec -ti -- journalctl -u uyuni-update-config
----

サーバコンテナを再起動する際に、このコマンドがすべての{salt}クライアントのpillarデータの更新をトリガーすることに注意してください。実行にかかる時間は、登録済みクライアントの数によって異なります。


=== 直接管理されているクライアントの再設定

*クライアントが{productname}プロキシを介して管理されている場合は、この手順をスキップします。*

次のプロシージャを使用して、直接管理されているクライアントを再設定し、新しいホスト名とIPアドレスを認識させるようにします。

.プロシージャ:直接管理されているクライアントの再設定
[role=procedure]
____

. すべてのクライアントの{salt}クライアント設定ファイルで、新しい{salt}マスター（{productname}サーバ）の名前を指定します。
  ファイル名は`/etc/venv-salt-minion/minion.d/susemanager.conf`です。もしくは、{salt}バンドルを使用しない場合は`/etc/salt-minion/minion.d/susemanager.conf`となります。

+

----
master: <new_hostname>
----

. すべてのクライアントで{salt}サービスを再起動します。
  次のいずれかを実行します:

+

[source,shell]
----
systemctl restart venv-salt-minion
----

+

または、{salt}バンドルを使用しない場合は、次のコマンドを実行します:

+

[source,shell]
----
systemctl restart salt-minion
----

____


=== highstateを適用したクライアント接続

最後に、ホスト名を{salt}クライアント設定に完全に伝播させるために、ハイステートを適用します。
ハイステートを適用すると、リポジトリのURLにホスト名が更新されます。



== プロキシの再設定

すべてのプロキシを再構成する必要があります。
新しいサーバー証明書とキーはプロキシにコピーする必要があります。
詳細については、xref:installation-and-upgrade:install-proxy.adoc[]を参照してください。


[IMPORTANT]
====
プロキシを介してPXEブートを使用する場合は、プロキシの設定を確認する必要があります。
非コンテナ化された {productname} プロキシ 4.3 を介して PXE ブートを使用する場合、`tftpsync` を再構成する必要があります。

コンテナホストで、次のコマンドを実行します。

[source,shell]
----
mgrctl exec -ti -- configure-tftpsync.sh
----
