:revdate: 2025-06-26

ifeval::[{uyuni-content} == true]

이 안내서에서는 {sl-micro} {microversion} 또는 {sles} {bci-mlm}에서 {productname} {productnumber} 프록시 컨테이너의 배포 절차를 설명합니다. 이 안내서에서는 {productname} {productnumber} 서버가 이미 배포된 상태를 가정합니다.

[IMPORTANT]
====
{sl-micro}는 당분간 일반 미니언([systemitem]``기본`` 연락 방법)으로만 지원됩니다. Salt SSH 클라이언트([systemitem]``salt-ssh`` 연락 방법)로도 관리할 수 있도록 개선하는 중입니다.
====


배포하려면 다음 작업을 수행합니다.


// Add certificate tasks item
.절차: 프록시 배포
[role=procedure]
_____
. 하드웨어 요구사항을 검토합니다.

. 서버에서 {sl-micro} {microversion} 또는 {sles} {bci-mlm} 상위 채널과 프록시 확장 하위 채널을 동기화합니다.

. 베어메탈 머신에 {sl-micro} 또는 {sles}를 설치합니다.

. Create a {salt} activation key with Proxy Extension.

. [systemitem]``기본`` 연결 방법을 사용하여 프록시를 클라이언트로 부트스트랩합니다.

. 프록시 구성을 생성합니다.

. 프록시 구성을 서버에서 프록시로 전송합니다.

. 프록시에 패키지를 설치합니다.

. 프록시 구성을 사용하여 클라이언트를 {productname}의 프록시로 등록합니다.

_____


.프록시 컨테이너 호스트에 지원되는 운영 체제
[NOTE]
====
컨테이너 호스트용으로 지원되는 운영 체제는 {sl-micro} {microversion} 및 {sles} {bci-mlm}입니다.

컨테이너 호스트::
컨테이너 호스트는 컨테이너를 관리하고 배포할 수 있는 컨테이너 엔진(예: Podman)이 탑재된 서버입니다. 이러한 컨테이너는 필수적인 부분(예: 애플리케이션과 라이브러리)을 보관하지만, 전체 운영 체제는 보관하지 않으므로 경량화됩니다. 이 설정은 애플리케이션이 다양한 환경에서 동일한 방식으로 실행되도록 보장합니다. 컨테이너 호스트는 이러한 컨테이너에 필요한 리소스(예: CPU, 메모리, 스토리지)를 제공합니다.
====


== 프록시의 하드웨어 요구사항

== 상위 및 프록시 확장 하위 채널 동기화

이 섹션에서는 서버 {webui}의 menu:관리자[설정 마법사 > 조직 자격 증명]에서 조직 자격 증명을 이미 입력한 것으로 가정합니다. 제품은 menu:관리자[설정 마법사 > 제품] 페이지에 나열되어 있습니다. 이 채널은 서버에서 완전히 동기화되어야 하며, 하위 채널 [systemitem]``프록시``가 확장 옵션으로 선택되어야 합니다.


.절차: 상위 채널 및 프록시 확장 동기화
[role=procedure]
_____

. {productname} {webui}에서 menu:관리[제품]으로 이동합니다.

. 제품 페이지에서 필터 필드에 {sl-micro} 또는 {sles}를 입력합니다.

. 다음으로 드롭다운을 사용하여 필요한 아키텍처를 선택합니다. 이 예제의 경우 {x86_64}입니다.

. [systemitem]``제품 설명`` 필드에서 {sl-micro} {microversion} 또는 {sles} {bci-mlm} 확인란을 선택한 후 드롭다운을 사용하여 [systemitem]``{productname} 프록시 확장 프로그램 5.1 x86_64`` 확장 프로그램을 선택합니다.

. btn:[제품 추가] 버튼을 클릭합니다.

. 동기화가 완료될 때까지 기다립니다.

_____

== 사용자 정의 영구 스토리지 구성

영구 스토리지 구성은 선택 사항이지만, 컨테이너 디스크가 가득 찬 상태에서 심각한 문제를 피할 수 있는 유일한 방법입니다. 인프라에 사용자 정의 영구 스토리지가 필요한 경우, [command]``mgr-storage-proxy`` 도구를 사용하십시오.

자세한 내용은 [command]``mgr-storage-proxy --help``를 참조하십시오. 이 도구는 컨테이너 스토리지 및 Squid 캐시 볼륨 생성을 간소화합니다.

다음 방법으로 명령 사용:

[source, shell]
----
mgr-storage-proxy <storage-disk-device>
----

예:

[source, shell]
----
mgr-storage-proxy /dev/nvme1n1
----



[NOTE]
====
이 명령은 [path]``/var/lib/containers/storage/volumes``에 영구 스토리지 볼륨을 생성합니다.

자세한 내용은 다음을 참조하십시오.

* xref:installation-and-upgrade:container-management/persistent-container-volumes.adoc[]
* xref:administration:troubleshooting/tshoot-container-full-disk.adoc[]
====

== {productname} 프록시 시작

이제 [literal]`mgrpxy` 명령으로 컨테이너를 시작할 수 있습니다.

[[proc-setup-containers-setup-start]]
.절차: 프록시 상태 시작 및 확인
[role=procedure]
_____

. 다음을 호출하여 프록시 시작:

+

----
mgrpxy start
----

. 다음을 호출하여 컨테이너 상태 확인:

+

----
mgrpxy status
----

+

5개의 {productname} 프록시 컨테이너가 있어야 하며, 이러한 컨테이너는 [literal]``proxy-pod`` 컨테이너 Pod에 속해야 합니다.

* proxy-salt-broker
* proxy-httpd
* proxy-tftpd
* proxy-squid
* proxy-ssh

== 서비스에 사용자 정의 컨테이너 이미지 사용

기본적으로 {productname} 프록시 제품군은 각 서비스에 대해 동일한 이미지 버전과 레지스트리 경로를 사용하도록 구성됩니다. 그러나 [literal]``-tag`` 및 [literal]``-image``로 끝나는 설치 파라미터를 사용하여 특정 서비스에 대한 기본값을 재정의할 수 있습니다.

예:

[source, shell]
----
mgrpxy install podman --httpd-tag 0.1.0 --httpd-image registry.opensuse.org/uyuni/proxy-httpd /path/to/config.tar.gz
----

이는 다시 시작하기 전에 [path]``registry.opensuse.org/uyuni/proxy-httpds``가 사용할 이미지이고 [literal]``0.1.0``이 버전 태그인 httpd 서비스의 구성 파일을 조정합니다.

값을 기본값으로 재설정하려면 해당 파라미터 없이 설치 명령을 다시 실행합니다.

[source, shell]
----
mgrpxy install podman /path/to/config.tar.gz
----

이 명령은 먼저 모든 서비스의 구성을 전역 기본값으로 재설정한 다음 다시 로드합니다.
