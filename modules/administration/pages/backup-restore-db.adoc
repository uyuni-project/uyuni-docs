[command]``mgradm`` is a {productname} utility used to perform online database backups via {postgresql} continuous archiving and Write-Ahead Logging (WAL), similar to [command]``smdba`` in older versions.

== Overview

Continuous archiving reduces the risk of data loss by constantly backing up the transaction logs (WAL) to a dedicated volume.

The backup system utilizes a dedicated Podman volume named [literal]``var-pgsql-walbackup`` to store:

* A base backup (a full snapshot of the database).
* WAL segments (incremental changes since the last base backup).

[NOTE]
====
Point-in-time recovery is currently not supported by [command]``mgradm`` and requires a manual workflow.


Only full recovery is supported by [command]``mgradm`` tool. 
The restore procedure will restore everything from the backup location.
====

== Prerequisites

* [command]``mgradm`` tool installed on the host.
* {productname} server running in a Podman environment.

== Enabling Backups

To enable continuous archiving, run the following command:

[source,shell]
----
mgradm backup db enable
----

.What this command does:
1. Verifies that the backup is not already enabled.
2. Configures {postgresql} ([path]``postgresql.conf``) to enable [option]``archive_mode`` and sets the [option]``archive_command``.
3. Configures the [literal]``var-pgsql-walbackup`` volume for the database service.
4. Restarts the database service to apply changes.
5. Performs an initial base backup inside the container.

If the backup is already configured but you need to re-initialize it, use the [option]``--force`` flag.

== Checking Backup Status

You can verify the current state of the database backup system at any time:

[source,shell]
----
mgradm backup db status
----

The command will report one of the following statuses:

* *enabled*: Backup is correctly configured and active.
* *disabled*: Continuous archiving is explicitly turned off.
* *misconfigured*: There is a discrepancy between the configuration and the runtime state (for example, missing volume, incorrect archive command).

== Maintaining Backups (Rebasing)

Over time, the number of WAL segments can grow significantly, consuming disk space and increasing restoration time.
It is recommended to periodically "rebase" the backup by creating a new base backup and starting a new WAL chain.

[source,shell]
----
mgradm backup db rebase
----

This command creates a fresh full snapshot of the database without requiring a service restart.

== Restoring from Backup

Restoring from a backup is a *destructive operation* that replaces the current database data with the content of the backup.

[source,shell]
----
mgradm backup db restore
----

.The restoration process:
1. Stops the database service.
2. Clears the current database data directory.
3. Extracts the base backup from the backup volume.
4. Configures {postgresql} for recovery.
5. Starts the database service.
6. Replays the WAL segments to bring the database to the latest possible state.

The command will wait until the database has fully recovered and is ready to accept connections.

[IMPORTANT]
====
Ensure you have a recent base backup and all necessary WAL segments in the [literal]``var-pgsql-walbackup`` volume before performing a restore.
====

== Disabling Backups

To turn off continuous archiving:

[source,shell]
----
mgradm backup db disable
----

By default, this command disables archiving in the configuration and restarts the database but preserves the backup volume.

To also remove the backup volume and all stored backups, use:

[source,shell]
----
mgradm backup db disable --purge-volume
----
