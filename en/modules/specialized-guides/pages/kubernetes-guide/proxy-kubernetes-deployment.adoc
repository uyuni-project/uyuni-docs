:revdate: 2026-06-24
:page-revdate: {revdate}

== Proxy on {kube} changes

There were multiple changes in how to install {productname} proxies running on {kube}:

* [command]``mgrpxy`` is no longer handling proxies on {kube}, helm and the [literal]``proxy-helm`` chart need to be used instead.

* The TLS certificates have to be in secrets, rather than in the configuration tarball.
  This aims at allowing cloud-native TLS certificates management for the proxies.

* The proxy queries the server at the start of the container to verify that the versions are compatible.

* The needed persistent volume claims has been reduced to the squid cache only.

ifeval::[{mlm-content} == true]
* The {productname} proxy is supported when running on an RKE2 or K3S cluster.
endif::[]



== Prerequisites

Installing the {kube} cluster and configuring it is out of the scope of this document.

The cluster is assumed to be ready to be used with a user having rights on a namespace dedicated to {productname}.

Create [literal]``Role`` and [literal]``RoleBinding`` if they do not exist already. 
The minimum rights required to deploy [literal]``proxy-helm`` are defined as:

[source,yaml]
----
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: example-resource-manager
  namespace: uyuni-proxy
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log", "services", "secrets", "configmaps", "persistentvolumeclaims"]
  verbs: ["*"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["*"]
- apiGroups: ["networking.k8s.io"]
  resources: ["ingresses"]
  verbs: ["*"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: example-resource-manager-binding
  namespace: uyuni-proxy
subjects:
- kind: User
  name: $USERNAME
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: example-resource-manager
  apiGroup: rbac.authorization.k8s.io
----


[IMPORTANT]
====
This guide assumes the reader knows how to work with {kube}: the concepts will not be explained here as they are extensively documented in the official {kube} documentation.
====

The {productname} administrator needs to deploy the [literal]``proxy-helm`` {helm} chart.
However, this chart requires to prepare:

* TLS certificates chain for the proxy,

* a [literal]``ConfigMap`` for the proxy root CA certificate,

* a persistent volumes  for the claim the chart will create or a storage class automatically creating it,

* Load balancers or other mechanisms to expose the Salt, SSH and TFTP ports.

[NOTE]
====
The TLS certificate and the root CA [literal]``ConfigMap`` are optional inputs: the [literal]``proxy-helm`` chart can create [literal]``proxy-cert`` and [literal]``uyuni-ca`` itself from the values carried in the configuration tarball.
See the TLS setup section below for the trade-offs and the adoption procedure for pre-existing objects.
====

Run the following command to read the full details on how to use the proxy {helm} chart:

ifeval::[{mlm-content} == true]
[source, Shell, subs="attributes"]
----
helm show readme --version {productchartversion} \
    oci://registry.suse.com/suse/multi-linux-manager/5.2/proxy-helm
----
endif::[]

ifeval::[{uyuni-content} == true]
[source, Shell, subs="attributes"]
----
helm show readme --version {productchartversion} \
    oci://registry.opensuse.org/uyuni/proxy-helm
----

[NOTE]
====
Helm requires the version to be semver2 compatible when using OCI repositories.
This is why the version ([literal]``{productchartversion}``) is formatted differently from the release number ([literal]``{productnumber}``).
====

For older versions, the Helm chart is located in a snapshot repository like: [literal]``oci://registry.opensuse.org/systemsmanagement/uyuni/snapshots/<version>/opensuse_tumbleweed/uyuni/proxy-helm``.
endif::[]




ifeval::[{mlm-content} == true]
=== Credentials

A secret with the SCC credentials needs to be defined in order to pull the images from `registry.suse.com`.
Refer to https://kubernetes.io/docs/tasks/configure-pod-container/pull-image-private-registry/ for the instructions to prepare the secret.
Set the `registrySecret` [literal]``proxy-helm`` chart value to the name of the secret containing those credentials to use it.
endif::[]



=== TLS setup

The proxy needs the following objects in its namespace:

* A TLS secret named [literal]``proxy-cert`` containing the proxy server certificate and key, with the public FQDN as Subject Alternate Name.

* A [literal]``ConfigMap`` named [literal]``uyuni-ca`` containing the root CA in the [literal]``ca.crt`` key.

In a cloud-native setup the certificates are typically managed externally (for example by [literal]``cert-manager``); create the objects beforehand:

[source, Shell]
----
kubectl create secret tls proxy-cert -n uyuni-proxy \
    --cert=proxy.crt --key=proxy.key

kubectl create configmap uyuni-ca -n uyuni-proxy \
    --from-file=ca.crt=/path/to/uyuni-ca.crt
----

The certificate file passed to [command]``kubectl create secret tls`` needs to start with the server certificate followed by the chain of intermediary CA certificates if any.
The root CA is not needed in this secret as it is expected in the [literal]``uyuni-ca`` [literal]``ConfigMap``.

[NOTE]
====
Both objects can also be created by the [literal]``proxy-helm`` chart itself from the values in the configuration tarball, with no manual [command]``kubectl create`` step:

* [literal]``proxy-cert`` is rendered from the [literal]``server_crt`` and [literal]``server_key`` fields of [literal]``httpd.yaml``.

* [literal]``uyuni-ca`` is rendered from the [literal]``ca_crt`` field of [literal]``config.yaml``.

This requires the configuration tarball to actually contain those fields.
If the proxy configuration was generated with the SSL part skipped (for example via the [guimenu]``SSL certificate`` selection list in the {webui}, or with [command]``spacecmd proxy_container_config_nossl``), the tarball carries no certificates and the auto-render path produces nothing — [literal]``proxy-cert`` and [literal]``uyuni-ca`` must then be created manually as shown above.

When [literal]``proxy-cert`` or [literal]``uyuni-ca`` already exist in the namespace (created manually or by a previous tool), the chart leaves them alone and the proxy uses the values already present.
To switch a pre-existing object over to helm management from the tarball values, set the [literal]``cert.takeOwnership`` chart value to [literal]``true`` and pass [command]``helm install --take-ownership`` ({helm} 3.14+) once:

[source, Shell, subs="attributes"]
----
helm upgrade --install uyuni-proxy \
    oci://registry.suse.com/suse/multi-linux-manager/5.2/proxy-helm \
    --version {productchartversion} \
    --namespace uyuni-proxy \
    --take-ownership \
    --set cert.takeOwnership=true \
    --set-file global.config=/root/proxy-config/config.yaml \
    --set-file global.httpd=/root/proxy-config/httpd.yaml \
    --set-file global.ssh=/root/proxy-config/ssh.yaml
----

The toggle is required because the default skip-if-exists behavior leaves nothing for [command]``--take-ownership`` to adopt.
After adoption the objects carry the helm [literal]``managed-by`` label and the chart treats them as helm-managed: value changes are applied on subsequent upgrades, and they are removed on [command]``helm uninstall`` like any other chart resource.
On later upgrades [literal]``cert.takeOwnership`` can be left at [literal]``true`` or dropped back to [literal]``false`` — the default render condition already covers helm-managed objects.
====



=== Storage

The proxy chart defines a volume as a Persistent Volume Claim (PVC).

[WARNING]
====
* The creation of the underlying PV is the responsibility of the cluster administrators.

* The PVC use the [literal]``ReadWriteOnce`` access mode.
====

The created PVC can be tuned {helm} chart values, it can have the following values:

* [literal]``size``: to set the requested size of the PVC.

* [literal]``storageClass``: can be used to select the storage class to use for the PVC.

* [literal]``extraLabels``: can be used to add custom labels to the PVC.

* [literal]``annotations``: can be used to set custom annotations on the PVC.

* [literal]``volumeName``: can be used to hard code which volume the PVC should be bound to.

* [literal]``selector``: is the YAML fragment of the PVC selector to use to find the PV to bind to.

Refer to https://kubernetes.io/docs/concepts/storage/persistent-volumes/ for more information on persistent volumes and their claims.


Refer to the [literal]``proxy-helm`` README for the list of persistent volume claims which will be created and will need to be bound to persistent volumes.


[IMPORTANT]
====
While the default sizes are provided, it is highly recommended to change them based on the distributions you plan to synchronize.

For more information on storage requirements see xref:installation-and-upgrade:general-requirements.adoc[].
====



=== Exposing ports

{productname} proxy requires some TCP and UDP ports to be routed to its services.
Refer to the [literal]``proxy-helm`` README for the list of ports to be exposed.

[WARNING]
====
RKE2 ships with nginx as the default ingress controller.
However, as this is deprecated and soon to be unsupported, the [literal]``proxy-helm`` chart defaults to use Traefik as ingress controller.
Using the nginx ingress controller might work and will not be documented, use at your own risk.

K3S ships Traefik by default, so no controller swap is needed.
====

[NOTE]
====
The [literal]``proxy-helm`` chart supports Gateway API version 1.4.
Since this requires experimental CRDs which are not shipped with RKE2 1.35 or the bundled K3S Traefik, it is not recommended to be used in production.
====

There are multiple ways to expose the ports, but this documentation will only mention how to configure the bundled Traefik on RKE2 and K3S.
This is not a task for the {productname} administrator, but the {kube} cluster administrator as it requires configuration to be set on the cluster nodes.

==== On RKE2

Create [literal]``/var/lib/rancher/rke2/server/manifests/uyuni-traefik.yaml`` on each node with the following content.
Note that Traefik takes a few seconds to be reinstalled after saving the file.

[source,yaml]
----
apiVersion: helm.cattle.io/v1
kind: HelmChartConfig
metadata:
  name: rke2-traefik
  namespace: kube-system
spec:
  valuesContent: |-
    ports:
      ssh:
        port: 8022
        expose:
          default: true
        exposedPort: 8022
        protocol: TCP
        hostPort: 8022
      salt-publish:
        port: 4505
        expose:
          default: true
        exposedPort: 4505
        protocol: TCP
        hostPort: 4505
        containerPort: 4505
      salt-request:
        port: 4506
        expose:
          default: true
        exposedPort: 4506
        protocol: TCP
        hostPort: 4506
        containerPort: 4506
----

==== On K3S

K3S ships Traefik in the [literal]``kube-system`` namespace. Create [literal]``/var/lib/rancher/k3s/server/manifests/uyuni-traefik.yaml`` on each node with the following content.
The [literal]``HelmChartConfig`` name is [literal]``traefik`` and [literal]``hostPort``/[literal]``containerPort`` are not required:

[source,yaml]
----
apiVersion: helm.cattle.io/v1
kind: HelmChartConfig
metadata:
  name: traefik
  namespace: kube-system
spec:
  valuesContent: |-
    ports:
      ssh:
        port: 8022
        exposedPort: 8022
        protocol: TCP
        expose:
          default: true
      salt-publish:
        port: 4505
        exposedPort: 4505
        protocol: TCP
        expose:
          default: true
      salt-request:
        port: 4506
        exposedPort: 4506
        protocol: TCP
        expose:
          default: true
----

[NOTE]
====
On a vanilla K3S install, the bundled Traefik picks up every [literal]``Ingress`` regardless of class.
Set [literal]``ingress.class: ""`` in the [literal]``proxy-helm`` values so the chart does not emit a class annotation that would be ignored.
If the Traefik install is customized to filter by class (for example via [literal]``--providers.kubernetescrd.ingressclass=traefik``), keep [literal]``ingress.class: "traefik"``.
====

[NOTE]
====
K3S's built-in load balancer ([literal]``klipper-lb`` / [literal]``serviceLB``) does not support the TFTP protocol.
Expose TFTP via the host network instead by setting [literal]``tftp.hostNetwork: true`` in the chart values.
====

[NOTE]
====
The Traefik entryPoint names ([literal]``ssh``, [literal]``salt-publish`` and [literal]``salt-request``) must match the [literal]``ingress.entryPoints`` values configured in the [literal]``proxy-helm`` chart (the names above are the chart defaults).
If the surrounding Traefik installation uses different entryPoint names, override the chart defaults via the [literal]``ingress.entryPoints`` value:

[source,yaml]
----
ingress:
  entryPoints:
    ssh: "uyuni-ssh"
    saltPublish: "uyuni-publish"
    saltRequest: "uyuni-request"
----
====

If Traefik is used as the Ingress controller, the user needs access to additional resources.
Add the following to the rules of the previously defined role:

[source,yaml]
----
- apiGroups: ["traefik.io", "traefik.containo.us"]
  resources: ["ingressroutetcps"]
  verbs: ["*"]
----

If Gateway API is used instead, add the following to the rules of the previously defined role:

[source,yaml]
----
- apiGroups: ["gateway.networking.k8s.io"]
  resources: ["gateways", "httproutes", "tcproutes"]
  verbs: ["*"]
----

TFTP is complex to expose from a {kube} pod due to the nature of the protocol: the TFTP server receives requests on port 69, but negotiates another random port to continue.
This port also needs to stay the same through the whole session for the server to recognize the client as being the same.
This means that there are only two possible ways to use the TFTP server:

* using a load balancer compatible with TFTP,

* using the host network for the TFTP pod. This can be achieved by setting the [literal]``tftp.hostNetwork`` helm chart value to [literal]``true``.



=== DNS configuration

The [literal]``dnsConfig`` chart value is a passthrough to the pod's https://kubernetes.io/docs/concepts/services-networking/dns-pod-service/#pod-dns-config[Kubernetes [literal]``PodDNSConfig``].
It is unset by default so the pod inherits the cluster's normal DNS behavior.

If external FQDN lookups from the proxy hang for around 10 seconds (the default [literal]``urlopen`` timeout), the cluster's default [literal]``ndots:5`` is likely expanding the name through several cluster search domains before falling back to the bare lookup.
Override with:

[source,yaml]
----
dnsConfig:
  options:
    - name: ndots
      value: "1"
----

== Deploying the {productname} Proxy Helm Chart

Copy and extract the generated configuration [literal]``tar.gz`` file and then install using helm:

ifeval::[{mlm-content} == true]
[source, Shell, subs="attributes"]
----
helm install smlm-proxy \
    oci://registry.suse.com/suse/multi-linux-manager/5.2/proxy-helm \
    -n uyuni-proxy \
    --description "Proxy installation" \
    --set "registrySecret=the-scc-secret" \
    --set-file global.config=path/to/config.yaml \
    --set-file global.ssh=path/to/ssh.yaml \
    --set-file global.httpd=path/to/httpd.yaml \
----
endif::[]

ifeval::[{uyuni-content} == true]
[source, Shell, subs="attributes"]
----
helm install uyuni-proxy \
    oci://registry.opensuse.org/uyuni/proxy-helm \
    -n uyuni-proxy \
    --description "Proxy installation" \
    --set-file global.config=path/to/config.yaml \
    --set-file global.ssh=path/to/ssh.yaml \
    --set-file global.httpd=path/to/httpd.yaml \
----
endif::[]

When setting multiple values, using a YAML values file is recommended instead of passing several [literal]``--set`` parameters.
Refer to the [command]``helm`` command help for more details.
