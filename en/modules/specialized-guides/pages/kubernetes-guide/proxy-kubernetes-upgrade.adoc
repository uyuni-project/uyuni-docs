= {productname} Proxy Upgrade on {kube}
:revdate: 2026-06-03
:page-revdate: {revdate}

Proxies deployed via the [literal]``proxy-helm`` chart (on either RKE2 or K3S) are upgraded by re-running [command]``helm upgrade --install`` against a newer chart version. The chart values and the existing PVC binding are preserved between releases.

[NOTE]
====
This is the supported {kube} upgrade path. The legacy [command]``mgrpxy install kubernetes`` / [command]``mgrpxy upgrade kubernetes`` flow on K3S is replaced by the direct chart-based deployment; if you are still on that path, follow the one-time migration in xref:kubernetes-guide/migrate-proxy-mgrpxy-to-helm.adoc[] before the procedure below applies.
====

.Procedure: Upgrade the proxy on {kube}
[role=procedure]
____

ifeval::[{mlm-content} == true]
. Run [command]``helm upgrade`` against the new chart version, reusing the values from the previous release:

+

[source,console,subs="attributes"]
----
helm upgrade uyuni-proxy \
    oci://registry.suse.com/suse/multi-linux-manager/{productnumber}/proxy-helm \
    --version {productchartversion} \
    --namespace uyuni-proxy \
    --reuse-values
----
endif::[]

ifeval::[{uyuni-content} == true]
. Run [command]``helm upgrade`` against the new chart version, reusing the values from the previous release:

+

[source,console,subs="attributes"]
----
helm upgrade uyuni-proxy \
    oci://registry.opensuse.org/uyuni/proxy-helm \
    --version {productchartversion} \
    --namespace uyuni-proxy \
    --reuse-values
----
endif::[]

+

[NOTE]
====
[literal]``--reuse-values`` keeps the values from the previous release intact ([literal]``global.config``/[literal]``global.httpd``/[literal]``global.ssh``, [literal]``ingress.class``, [literal]``tftp.hostNetwork``, [literal]``volumes.squid.volumeName``, [literal]``dnsConfig``, [literal]``registrySecret``, etc.), so no [literal]``--set`` / [literal]``--set-file`` flags need to be re-passed for a routine version bump.

If the proxy configuration tarball changed since the last install (TLS certificate rotation, server FQDN change, re-registration), re-extract the files and overlay them on top of [literal]``--reuse-values``:

[source,console]
----
mkdir -p /root/proxy-config
kubectl -n uyuni-proxy exec deploy/uyuni-proxy -c httpd -- cat /etc/uyuni/config.yaml > /root/proxy-config/config.yaml
kubectl -n uyuni-proxy exec deploy/uyuni-proxy -c httpd -- cat /etc/uyuni/httpd.yaml > /root/proxy-config/httpd.yaml
kubectl -n uyuni-proxy exec deploy/uyuni-proxy -c ssh   -- cat /etc/uyuni/ssh.yaml   > /root/proxy-config/ssh.yaml
----

Then add the matching [literal]``--set-file global.config=...`` / [literal]``--set-file global.httpd=...`` / [literal]``--set-file global.ssh=...`` flags to the [command]``helm upgrade`` command above.
The same pattern applies to any other value that needs to change: keep [literal]``--reuse-values`` and overlay the override with [literal]``--set``.
====

+

. Watch the rollout and verify the pods come back up:

+

[source,console]
----
kubectl -n uyuni-proxy rollout status deploy/uyuni-proxy
kubectl -n uyuni-proxy get pod
----

____

On {kube} the image cleanup is handled automatically, or it depends on the {kube} distribution.
