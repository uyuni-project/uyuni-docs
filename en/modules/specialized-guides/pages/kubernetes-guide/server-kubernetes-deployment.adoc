[[deploy-server-on-kubernetes]]
= Server Deployment on {kube}
:revdate: 2026-02-24
:page-revdate: {revdate}


{productname} can be deployed on {kube} using a {helm} chart.
This guide covers deploying {productname} {productnumber} on RKE2.

== Pre-Deployment Checklist

Before proceeding, ensure your environment meets these requirements:

[%interactive]
* [ ] *Kubernetes Distribution:* RKE2 (other distributions are not supported).
* [ ] *Ingress Controller:* Traefik (RKE2 defaults to nginx, which is not supported).
* [ ] *Tools:* {helm} v3 and [literal]``kubectl`` configured with access to the cluster.
* [ ] *Permissions:* [literal]``cluster-admin`` privileges (or access to someone who has them).
* [ ] *Node-Level Access:* Root access to RKE2 nodes to apply AppArmor/SELinux policies and configure Traefik.
* [ ] *Storage:* A [literal]``StorageClass`` supporting [literal]``ReadWriteOnce`` volumes.

[NOTE]
====
If you do not have node-level access to apply security policies, or if your environment is not RKE2, you may want to consider a standard installation on a virtual machine or bare-metal instead.
Deploying on a {kube} cluster is still possible, but would require to run the main container as super privileged.
====

Several personas are involved in the installation of {productname} on {kube}:

* the {kube} administrator manages the cluster, its users and accesses,

* the infrastructure administrator takes care of wiring the network access to the cluster,

* the PKI administrator is responsible for the TLS certificates generation and deployment infrastructure,

* the {productname} administrator controls the application itself and its deployment.

In some cases those personas can be merged into a single person or team, but keeping them in mind explains why the installation is not a one-shot script.
{kube} clusters can vary a lot between organizations, so the {productname} core installation is designed to be as agnostic as possible of those specifics.



== Supported configuration

[cols="1,3", options="header"]
|===
| Component | Supported

| {kube} distribution
| RKE2.
Other distributions are not tested and not supported.

| RKE2 version
| Refer to the {productname} release notes for the minimum supported RKE2 version.

| Ingress controller
| Traefik.
The nginx ingress controller shipped with RKE2 is deprecated and will be removed in a future RKE2 release; it is not documented here, nor tested.

| Gateway API
| Supported from version 1.4, but requires experimental CRDs not included in RKE2 by default.
Not recommended for production use.

| Storage
| Any storage class providing [literal]``ReadWriteOnce`` volumes.
RKE2 ships without any provisioner.
The [literal]``local-path`` provisioner is suitable for testing but provides no redundancy and is not recommended for production.

| CNI
| No special requirements.
Canal (the RKE2 default) is supported.

| Required tools
| {helm} v3 and [literal]``kubectl``, configured to access the target cluster.
|===

[NOTE]
====
Because the main server container runs [command]``systemd``, {productname} on {kube} is not fully cloud-native.
Running it requires elevated security permissions that standard {kube} security profiles do not allow by default.
See <<security-framework-setup>> for details.
====



== Preparation

[IMPORTANT]
====
This guide assumes the reader knows how to work with {kube}: the concepts will not be explained here as they are extensively documented in the official {kube} documentation.
====

The {productname} administrator needs to deploy the [literal]``server-helm`` {helm} chart.
However, this chart requires preparing:

* TLS certificates chain for the server and database,

* [literal]``ConfigMaps`` for the server and database root CA certificates,

* persistent volumes  for the claims the chart will create or a storage class automatically creating them,

* credentials secrets for the database users and the administrator,

* Load balancers or other mechanisms to expose the Salt, report database and optionally TFTP ports.

Run the following command to read the full details on how to use the server {helm} chart:

ifeval::[{mlm-content} == true]
[source, Shell, subs="attributes"]
----
helm show readme --version {productchartversion} \
    oci://registry.suse.com/suse/multi-linux-manager/5.2/server-helm
----
endif::[]

ifeval::[{uyuni-content} == true]
[source, Shell, subs="attributes"]
----
helm show readme --version {productchartversion} \
    oci://registry.opensuse.org/uyuni/server-helm
----

[NOTE]
====
Helm requires the version to be semver2 compatible when using OCI repositories.
This is why the version ([literal]``{productchartversion}``) is formatted differently from the release number ([literal]``{productnumber}``).
====

For older versions, the Helm chart is located in a snapshot repository like: [literal]``oci://registry.opensuse.org/systemsmanagement/uyuni/snapshots/<version>/opensuse_tumbleweed/uyuni/server-helm``.
endif::[]



[[namespace]]
=== Namespace

Install {productname} into a dedicated namespace named [literal]``uyuni-server``.
This keeps the server workload clearly separated from other workloads on the cluster, and allows support tooling to collect logs without requiring manual input.
The companion proxy guide uses [literal]``uyuni-proxy`` by the same convention.

[IMPORTANT]
====
If you choose a different namespace, use it consistently throughout all commands in this guide.
ifeval::[{mlm-content} == true]
Support tooling assumes the [literal]``uyuni-server`` namespace by default.
endif::[]
====

Create the namespace and set the variable used throughout this guide:

[source,shell]
----
kubectl create namespace uyuni-server
NAMESPACE=uyuni-server
----



=== User permissions

[NOTE]
====
If the {productname} administrator already holds [literal]``cluster-admin`` permissions, the RBAC setup below can be skipped.
====

The user installing the {productname} server {helm} chart needs to have permissions on the namespace.
The following [literal]``Role`` and [literal]``RoleBinding`` grant the minimum rights required to deploy [literal]``server-helm`` with Traefik:

[source,yaml]
----
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: uyuni-server-manager
  namespace: uyuni-server # <1>
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log", "services", "secrets", "configmaps", "persistentvolumeclaims"]
  verbs: ["*"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["*"]
- apiGroups: ["networking.k8s.io"]
  resources: ["ingresses"]
  verbs: ["*"]
- apiGroups: ["traefik.io", "traefik.containo.us"]
  resources: ["ingressroutetcps", "middlewares"]
  verbs: ["*"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: uyuni-server-manager-binding
  namespace: uyuni-server # <1>
subjects:
- kind: User
  name: REPLACE_WITH_USERNAME # <2>
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: uyuni-server-manager
  apiGroup: rbac.authorization.k8s.io
----
<1> Replace with your namespace if you chose a different name.
<2> Replace with the actual {kube} username of the {productname} administrator.

Save this to a file such as [filename]``uyuni-server-rbac.yaml``, then apply it as the cluster administrator:

[source,shell]
----
kubectl apply -f uyuni-server-rbac.yaml
----

=== Credentials

The deployment requires four specific secrets of [literal]``kubernetes.io/basic-auth`` type, each containing a [literal]``username`` and a [literal]``password`` key.
The following commands use the [literal]``$NAMESPACE`` variable set in the <<namespace>> section.
Replace [literal]``supersecret`` with actual passwords before running them.

[IMPORTANT]
====
Using [literal]``--from-literal`` for the passwords is not a secure practice, read the command help to use [literal]``--from-file`` or [literal]``--from-env-file``.
If using declarative YAML definitions instead of these commands, make sure these files are encrypted before pushing them to a remote version control.
====

[source, Shell]
----
kubectl create secret generic -n $NAMESPACE --type 'kubernetes.io/basic-auth' \
    --from-literal=username=dbadmin \
    --from-literal=password=supersecret \
    db-admin-credentials
kubectl create secret generic -n $NAMESPACE --type 'kubernetes.io/basic-auth' \
    --from-literal=username=dbuser \
    --from-literal=password=supersecret \
    db-credentials
kubectl create secret generic -n $NAMESPACE --type 'kubernetes.io/basic-auth' \
    --from-literal=username=reportdb \
    --from-literal=password=supersecret \
    reportdb-credentials
kubectl create secret generic -n $NAMESPACE --type 'kubernetes.io/basic-auth' \
    --from-literal=username=admin \
    --from-literal=password=supersecret \
    admin-credentials
----


ifeval::[{mlm-content} == true]
{productname} images are hosted on [literal]``registry.suse.com`` and require authentication.
Create an image pull secret using your SUSE Customer Center (SCC) mirroring credentials.
Mirroring credentials can be found at https://scc.suse.com under menu:My Organization[Mirroring Credentials].

[source,shell]
----
kubectl create secret docker-registry scc-credentials \
    -n $NAMESPACE \
    --docker-server=registry.suse.com \
    --docker-username=YOUR_SCC_MIRRORING_USERNAME \
    --docker-password=YOUR_SCC_MIRRORING_PASSWORD
----

[NOTE]
====
SCC mirroring credentials are different from your regular SCC login.
Use the dedicated mirroring username and password shown on the Mirroring Credentials page, not your SCC account email and password.
====

Pass the secret name to the [literal]``server-helm`` chart during installation:

[source,shell]
----
--set "registrySecret=scc-credentials"
----
endif::[]



=== TLS setup

The chart expects two TLS secrets and two ConfigMaps holding the corresponding root CA certificates.

[literal]``uyuni-cert``::
TLS certificate for the Ingress rule.
Must include the public FQDN as a Subject Alternate Name.

[literal]``db-cert``::
TLS certificate for the report database.
Must include the public FQDN if the report database is accessed from outside the cluster. It can be the same
certificate as [literal]``uyuni-cert``.

[NOTE]
====
The certificate file must start with the server certificate followed by any intermediate CA certificates.
Do not include the root CA in the [literal]``uyuni-cert`` secret — root CAs go into ConfigMaps (see below).
====

Create the server ingress certificate secret:

[source,shell]
----
kubectl create secret tls uyuni-cert \
    -n $NAMESPACE \
    --cert=/path/to/server.crt \
    --key=/path/to/server.key
----

The report database secret requires an additional [literal]``ca.crt`` key that [command]``kubectl create secret tls`` does not produce.
Create it as a generic secret instead:

[source,shell]
----
kubectl create secret generic db-cert \
    -n $NAMESPACE \
    --from-file=tls.crt=/path/to/db.crt \
    --from-file=tls.key=/path/to/db.key \
    --from-file=ca.crt=/path/to/db-ca.crt
----

[NOTE]
====
For self-signed certificates where the certificate is also its own CA, pass the same file for both [literal]``tls.crt`` and [literal]``ca.crt``.
====

Create ConfigMaps for the root CA certificates:

[source,shell]
----
kubectl create configmap uyuni-ca \
    -n $NAMESPACE \
    --from-file=ca.crt=/path/to/server-ca.crt

kubectl create configmap db-ca \
    -n $NAMESPACE \
    --from-file=ca.crt=/path/to/db-ca.crt
----

[TIP]
====
If you want to automate certificate issuance and renewal, [literal]``cert-manager`` together with [literal]``trust-manager`` can manage all of the above automatically.
See <<example-helm-charts>> for links to example charts that demonstrate this approach.
====



=== Storage

The server chart defines volumes as Persistent Volume Claims (PVCs).

Before proceeding, verify that a StorageClass is available on the cluster:

[source,shell]
----
kubectl get storageclass
----

If the output shows no resources, a provisioner must be installed before the chart can be deployed.
For testing, the local-path provisioner is a simple option:

[source,shell]
----
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/master/deploy/local-path-storage.yaml
----

[WARNING]
====
The local-path provisioner stores data on the node's local filesystem and provides no redundancy.
It is suitable for testing only.
For production, use a StorageClass backed by a proper storage system.
====

[WARNING]
====
* The creation of the underlying PVs is the responsibility of the cluster administrators.

* The PVCs use the [literal]``ReadWriteOnce`` access mode.
====

The created PVCs can be tuned using {helm} chart values.
Each of the PVCs can have the following values:

* [literal]``size``: to set the requested size of the PVC.

* [literal]``storageClass``: can be used to select the storage class to use for the PVC.
  This can be useful to select faster storage for the database or the packages storage.

* [literal]``extraLabels``: can be used to add custom labels to the PVC.

* [literal]``annotations``: can be used to set custom annotations on the PVC.

* [literal]``volumeName``: can be used to hard code which volume the PVC should be bound to.

* [literal]``selector``: is the YAML fragment of the PVC selector to use to find the PV to bind to.

Refer to https://kubernetes.io/docs/concepts/storage/persistent-volumes/ for more information on persistent volumes and their claims.

The following PVCs will be created by the chart:

[cols="2,1", options="header"]
|===
| PVC | Default size

| [literal]``var-pgsql``
| 50Gi

| [literal]``var-spacewalk``
| 100Gi

| [literal]``var-cache``
| 10Gi

| [literal]``var-log``
| 2Gi

| [literal]``var-search``
| 10Gi

| [literal]``srv-www``
| 100Gi

| [literal]``srv-tftpboot``
| 300Mi

| [literal]``ca-certs``
| 10Mi

| [literal]``etc-apache2``, [literal]``etc-cobbler``, [literal]``etc-postfix``, [literal]``etc-rhn``, [literal]``etc-salt``, [literal]``etc-sssd``, [literal]``etc-systemd-multi``, [literal]``etc-systemd-sockets``, [literal]``etc-tomcat``
| 1Mi each

| [literal]``etc-sysconfig``
| 20Mi

| [literal]``run-salt-master``, [literal]``srv-formulametadata``, [literal]``srv-pillar``, [literal]``srv-salt``, [literal]``srv-spacewalk``, [literal]``srv-susemanager``, [literal]``var-cobbler``, [literal]``var-salt``
| 1–10Mi each

|===

[IMPORTANT]
====
The default sizes for [literal]``var-spacewalk`` and [literal]``srv-www`` (100Gi each) are a starting point only.
Size them based on the number of distributions and packages you plan to synchronize.
For more information see xref:installation-and-upgrade:general-requirements.adoc[].
====

Override PVC settings in a values file rather than on the command line.
The following example increases the package and database volumes and pins them to a faster storage class:

[source,yaml]
----
volumes:
  var-spacewalk:
    size: 500Gi
    storageClass: fast-ssd
  srv-www:
    size: 500Gi
    storageClass: fast-ssd
  var-pgsql:
    size: 80Gi
    storageClass: fast-ssd
----

Pass the file to [command]``helm install`` with [option]``-f values.yaml``.



=== Installing Traefik

RKE2 ships with nginx as the default ingress controller.
The [literal]``server-helm`` chart requires Traefik, so nginx must be disabled and Traefik installed before proceeding.
This is a task for the {kube} cluster administrator.

Disable the nginx ingress controller by adding the following to [filename]``/etc/rancher/rke2/config.yaml`` on each RKE2 server node (create the file if it does not exist):

[source,yaml]
----
disable:
  - rke2-ingress-nginx
----

Install Traefik by creating [filename]``/var/lib/rancher/rke2/server/manifests/traefik.yaml`` on the RKE2 server node with the following content:

[source,yaml]
----
apiVersion: helm.cattle.io/v1
kind: HelmChart
metadata:
  name: traefik
  namespace: kube-system
spec:
  chart: traefik
  repo: https://traefik.github.io/charts
  targetNamespace: kube-system
----

[IMPORTANT]
====
The [literal]``metadata.name`` must be [literal]``traefik``.
The [literal]``server-helm`` chart creates Ingress resources with [literal]``ingressClassName: traefik``, which must match the IngressClass name created by the Helm release.
Using a different name (such as [literal]``rke2-traefik``) causes Traefik to silently ignore all {productname} ingresses, resulting in 404 responses even when all pods are running.
====

Restart RKE2 to apply both changes:

[source,shell]
----
systemctl restart rke2-server
----

Wait for Traefik to become ready before continuing

[[exposing-ports]]
=== Exposing ports

{productname} requires some TCP and UDP ports to be routed to its services.
Refer to the [literal]``server-helm`` README for the list of ports to be exposed.

[NOTE]
====
The [literal]``server-helm`` chart supports Gateway API version 1.4.
Since this requires experimental CRDs which are not shipped with RKE2 by default, it is not recommended for production use.
====

There are multiple ways to expose the ports, but this documentation will only cover how to configure RKE2's Traefik for this.
This is a task for the {kube} cluster administrator, not the {productname} administrator.

To set Traefik to expose and route the needed ports, create [filename]``/var/lib/rancher/rke2/server/manifests/uyuni-traefik.yaml`` on RKE2 server node (control plane node) with the following content.
Agent (worker) nodes do not have this directory and do not need the file.
Traefik takes a few seconds to be reconfigured after the file is saved.

[source,yaml]
----
apiVersion: helm.cattle.io/v1
kind: HelmChartConfig
metadata:
  name: traefik
  namespace: kube-system
spec:
  valuesContent: |-
    ports:
      reportdb-pgsql:
        port: 5432
        expose:
          default: true
        exposedPort: 5432
        protocol: TCP
        hostPort: 5432
        containerPort: 5432
      salt-publish:
        port: 4505
        expose:
          default: true
        exposedPort: 4505
        protocol: TCP
        hostPort: 4505
        containerPort: 4505
      salt-request:
        port: 4506
        expose:
          default: true
        exposedPort: 4506
        protocol: TCP
        hostPort: 4506
        containerPort: 4506
----

When using the server as a TFTP server there are a few issues to consider.
TFTP is complex to expose from a {kube} pod due to the nature of the protocol: the TFTP server receives requests on port 69, but negotiates another random port to continue.
This port also needs to stay the same through the whole session for the server to recognize the client as being the same.
This means that there are only two possible ways to use the TFTP server:

* using a load balancer compatible with TFTP,

* using the host network for the TFTP pod. This can be achieved by setting the [literal]``tftp.hostNetwork`` helm chart value to [literal]``true``.

[[security-framework-setup]]
=== Security framework setup

The main container of the server runs [command]``systemd``, which requires elevated privileges that the default {kube} security profiles do not allow.
There are two ways to handle this:

Super-privileged mode::
Set the [literal]``server.superPrivileged`` helm value to [literal]``true``.
This disables all Linux security module enforcement (SELinux, AppArmor) for the container and grants it broad host access.
+
[WARNING]
====
Super-privileged mode is the simplest option but significantly reduces the security posture of the node.
Use it only for testing and development, not for production deployments.
====

Custom AppArmor or SELinux profile (recommended for production)::
Create a custom policy or profile that grants only the specific permissions [command]``systemd`` requires.
This maintains security module enforcement while allowing the container to function correctly.

==== AppArmor

Obtain the profile content from the [literal]``server-helm`` chart README, which includes a ready-to-use profile named [literal]``k8s-systemd-uyuni``.
Deploy it on each cluster node, then load it:

[source,shell]
----
cp /path/to/k8s-systemd-uyuni /etc/apparmor.d/k8s-systemd-uyuni
apparmor_parser -r /etc/apparmor.d/k8s-systemd-uyuni
----

Then reference it in the helm values:

[source,yaml]
----
server:
  apparmorProfile: k8s-systemd-uyuni
----

==== SELinux

Obtain the policy file ([filename]``systemdcontainerpolicy.te``) from the [literal]``server-helm`` chart README.
Copy it to each cluster node, then compile and install it:

[source,shell]
----
checkmodule -M -m -o systemdcontainerpolicy.mod systemdcontainerpolicy.te
semodule_package -o systemdcontainerpolicy.pp -m systemdcontainerpolicy.mod
semodule -i systemdcontainerpolicy.pp
----

No helm value is required for SELinux — applying the policy module on the node is sufficient.


== Installation

The [literal]``server-helm`` chart requires one value to be set: [literal]``global.fqdn``.
The other values have sensible defaults, refer to the [literal]``server-helm`` chart README for more details on those.

The server can be installed with a command like the following:

ifeval::[{mlm-content} == true]
[source, Shell, subs="attributes"]
----
helm install smlm-server \
     oci://registry.suse.com/suse/multi-linux-manager/5.2/server-helm \
     -n $NAMESPACE \
     --description "Server installation" \
     --set "global.fqdn=the.server.fqdn" \
     --set "registrySecret=scc-credentials" \
     --version {productchartversion}
----
endif::[]

ifeval::[{uyuni-content} == true]
[source, Shell, subs="attributes"]
----
helm install uyuni-server \
     oci://registry.opensuse.org/uyuni/server-helm \
     -n $NAMESPACE \
     --description "Server installation" \
     --set "global.fqdn=the.server.fqdn" \
     --version {productchartversion}
----
endif::[]

When setting multiple values, using a YAML values file is recommended instead of passing several [literal]``--set`` parameters.
Refer to the [command]``helm`` command help for more details.



== Verifying the installation

After running [command]``helm install``, the server pod starts [command]``systemd`` and multiple services inside it.
This takes longer than a typical container startup — allow a few minutes before expecting the pod to be fully ready.

Watch the pods come up:

[source,shell]
----
kubectl get pods -n uyuni-server -w
----

All pods should eventually reach [literal]``Running`` status with all containers ready.
If a pod is stuck in [literal]``Pending``, [literal]``CrashLoopBackOff``, or [literal]``Error``, check its events:

[source,shell]
----
kubectl describe pod -n uyuni-server <POD_NAME>
kubectl logs -n uyuni-server <POD_NAME>
----

Once all pods are ready, open a browser and navigate to [literal]``https://<your-fqdn>``.
The {productname} setup wizard should appear.

To get a terminal inside the server container (equivalent to [command]``mgrctl term`` used elsewhere in the documentation):

[source, Shell]
----
SERVER_POD=`kubectl get pod -n uyuni-server -lapp.kubernetes.io/part-of=uyuni,app.kubernetes.io/component=server -o "jsonpath={.items[0].metadata.name}"`
kubectl exec -ti -n uyuni-server $SERVER_POD -- bash
----

== Troubleshooting

Start by checking the overall pod status and recent events:

[source,shell]
----
kubectl get pods -n uyuni-server
kubectl get events -n uyuni-server --sort-by='.lastTimestamp'
----

=== Pod stuck in Pending

A pod stays in [literal]``Pending`` when it cannot be scheduled.
The most common cause is an unbound PVC — the storage class has not provisioned a volume yet.

[source,shell]
----
kubectl get pvc -n uyuni-server
kubectl describe pvc -n uyuni-server <PVC_NAME>
----

Check that the storage class exists and is available, and that the PVC size does not exceed what the provisioner can provide.

=== Pod in CrashLoopBackOff

A pod that crashes immediately after starting is often caused by a missing secret or a security profile blocking [command]``systemd``.

Check the pod logs:

[source,shell]
----
kubectl logs -n uyuni-server <POD_NAME> --previous
----

If the log shows permission denied errors related to cgroup or mount operations, the AppArmor or SELinux profile is not applied correctly.
See <<security-framework-setup>> for setup instructions.

If the log shows secret or credential errors, verify that all required secrets exist in the namespace:

[source,shell]
----
kubectl get secrets -n uyuni-server
----

=== Web UI not reachable

If all pods are running but the web interface is not accessible, the ingress or port exposure is likely misconfigured.

Verify the Ingress resource was created:

[source,shell]
----
kubectl get ingress -n uyuni-server
----

Check that the FQDN in the Ingress matches the value passed to [literal]``global.fqdn`` during installation, and that DNS resolves to the cluster.
Also confirm that the Traefik configuration was applied on the RKE2 server nodes as described in <<exposing-ports>>.

=== 502 Bad Gateway

If the web interface returns a 502 Bad Gateway error and all pods are running, firewalld may be blocking traffic between the cluster's internal networks.
Add the pod and service networks to the trusted zone:

[source,shell]
----
firewall-cmd --zone=trusted --add-source=10.42.0.0/16 --permanent
firewall-cmd --zone=trusted --add-source=10.43.0.0/16 --permanent
firewall-cmd --reload
----

[NOTE]
====
The addresses above are the RKE2 defaults.
If the cluster was configured with custom pod or service CIDRs, use those instead.
====

ifeval::[{mlm-content} == true]
=== Image pull errors

If pods fail with [literal]``ImagePullBackOff``, the SCC credentials secret is missing or incorrect.

[source,shell]

Look for [literal]``unauthorized`` or [literal]``authentication required`` in the events.

=== Salt minions cannot connect

If the server is running but Salt minions cannot reach it, the TCP ports 4505 and 4506 are not exposed correctly.
Verify the Traefik configuration was applied on the RKE2 server nodes and that no firewall is blocking those ports between the minions and the cluster.
